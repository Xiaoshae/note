# 云计算

关于云计算概念的起源，工业界和学术界有不同的里程碑，但公认的现代云计算概念的普及主要归功于**Google（谷歌）**，而标准定义则归功于**NIST（美国国家标准与技术研究院）**。

- **最早的学术/商业萌芽 (1990s - 2006)：** 早在1996年，康柏（Compaq）公司的内部文件中就出现了“Cloud Computing”的字样。1997年，学者Ramnath Chellappa也使用了这个词。但是，当时并没有引起广泛关注。

- **现代概念的普及 (2006)：** 真正将云计算作为一种新型商业模式和技术愿景推向世界的是 **Google**。2006年8月9日，Google当时的首席执行官 **埃里克·施密特 (Eric Schmidt)** 在搜索引擎战略大会（Search Engine Strategies Conference）上首次在现代背景下提出了“Cloud Computing”的概念。他形容这是一种数据和服务都存储在服务器上的新兴模式。

  ​	*与此同时，Amazon（亚马逊）推出了AWS EC2，真正让云计算变成了可用的商业产品。*

- **权威定义的制定 (2011)：** 虽然Google和Amazon推广了技术，但**美国国家标准与技术研究院 (NIST)** 发布的 **SP 800-145** 文档，被全球公认为云计算的**标准定义**来源。



## 云计算定义

NIST 对云计算的定义非常严谨，它包含“**五大基本特征**”、“**三种服务模式**”和“**四种部署模型**”。

云计算是一种模型，它可以实现随时随地、便捷地、**按需**地通过**网络访问**一个可配置的**计算资源共享池**（例如网络、服务器、存储、应用和服务）。这些资源能够被**快速提供**和释放，且只需极少的管理工作或与服务提供商的交互。



**五大基本特征**

- **按需自助服务：** 用户可以根据需要自动获取计算能力（如服务器时间、网络存储），无需与服务商的人员进行人工交互。
- **广泛的网络访问：** 服务通过网络提供，并采用标准机制，支持各种客户端（手机、平板、笔记本、工作站）接入。
- **资源池化：** 供应商的计算资源被集中起来服务于多个消费者（多租户模式），根据需求动态分配物理和虚拟资源。用户通常不知道资源的具体地理位置。
- **快速弹性：** 资源可以快速、弹性地提供（有时是自动的），以快速扩展；在需求降低时也可以快速释放以缩小规模。
- **可计量的服务：** 云系统通过计量能力（如存储、处理能力、带宽、活跃账户数）自动控制和优化资源使用。



**三种服务模式**

- **IaaS (Infrastructure as a Service - 基础设施即服务)：**此时云就像一个“硬件出租商”。它提供虚拟的计算机、网络和存储。
- **PaaS (Platform as a Service - 平台即服务)：**此时云提供了一个“开发环境”。它帮你管理了操作系统和中间件，你只需要上传代码。
- **SaaS (Software as a Service - 软件即服务)：**此时云提供完整的“软件应用”。你只需要使用，什么都不用管。



**四种部署模型**

- **公有云：** 面向大众开放（如 AWS, Azure）。
- **私有云：** 专供某个组织使用，数据更安全。
- **混合云：** 结合了公有云和私有云，允许数据和应用在两者间移动。
- **社区云：** 由具有共同关注点（如任务、安全要求、政策）的特定社区共享。



## 云原生

要理解云原生，我们先看它的反面，云移民。



**云移民，又称云托管模式。** 

将基于传统物理数据中心环境构建的**单体架构**应用，在不重构代码逻辑、不改变系统架构的前提下，直接部署至云基础设施上。

应用仅仅是更换了运行载体（从物理机迁移至云服务器），其内部组件依然紧密耦合，状态管理与基础设施深度绑定。

由于未针对云环境进行适配，此类应用难以实现细粒度的水平扩展，故障隔离能力较弱，且无法利用云平台的自动化运维能力。其资源利用率和迭代效率往往受限于传统架构的瓶颈。



**云原生**

云原生就是**一种应用程序的设计哲学**（怎么写代码、怎么部署，才能最大限度地榨取云的价值），是利用云的特性（弹性、自动化、分布式），来构建和运行应用程序的方法论。

在应用设计之初即遵循云环境的动态特性，程序被设计成轻量、模块化、能随时重生。采用**微服务**架构将复杂系统解耦为独立的功能模块，并利用**容器化**技术实现环境的一致性与隔离性。



云原生计算基金会（CNCF）是这个领域的权威。云原生之所以厉害，是因为它依靠四大技术支柱：

**微服务**：云原生应用采用松耦合的架构设计，将单一应用拆分为一组小型的、功能单一的服务单元。每个服务运行在独立的进程中，拥有独立的数据存储，并可以由不同的团队使用不同的语言开发。服务之间通过轻量级的标准协议（如 HTTP/REST API 或 gRPC）进行通信。

**容器化**：容器化技术实现了一种不可变基础设施的交付形态。它将应用程序及其所有依赖项（代码、运行时、系统工具、库、配置）打包到一个标准的、轻量级的镜像中。容器在任何支持容器运行时的基础设施（无论是公有云、私有云还是本地开发机）上，都能保证运行环境的完全一致，实现了“一次构建，到处运行”。

**容器编排**：Kubernetes 是一个用于自动部署、扩展和管理容器化应用程序的开源平台。它基于声明式配置也就是用户定义“期望状态”，系统自动达成该状态。

**DevOps 与持续交付 (CI/CD)**：通过DevOps文化打破开发与运维的壁垒，并依托**持续集成（CI）和持续交付/部署（CD）**的自动化流水线。开发人员提交代码后，系统自动触发构建、单元测试、集成测试、安全扫描，并自动部署到测试或生产环境。



## DevOps

DevOps 是目前软件工程领域最重要的核心理念之一。它的出现彻底改变了软件生产的方式。

要理解 DevOps，我们不能只把它看作是一套工具（比如 Jenkins 或 Docker），它实际上是一种**文化变革**、**流程优化**和**技术实践**的结合体。



在 DevOps 出现之前，很多公司的 IT 部门存在一堵无形的“墙”，将两个团队隔离开来：

**开发团队 (Dev - Development)：**

- **目标：** 求变。他们想不断开发新功能，发布新版本。
- **KPI：** 发布频率、新功能数量。

**运维团队 (Ops - Operations)：**

- **目标：** 求稳。他们希望服务器永远不宕机，系统永远不出错。
- **KPI：** 系统可用性、无故障运行时间。



开发人员为了赶进度，把写好的代码直接“扔过墙”给运维。运维拿到代码，发现跑不起来（环境不一致），或者上线后系统崩了。运维怪开发代码烂，开发怪运维环境配置不对。这被称为**“混乱之墙”**。**DevOps 的诞生，就是为了推倒这堵墙。**



### 定义

**DevOps** 是 **Development（开发）** 和 **Operations（运维）** 两个词的组合。**DevOps 是一个“三位一体”的混合概念。**

它**首先是一种文化（思想）**，**其次是一套方法论（流程）**，**最后才是一组工具链（技术）**。

DevOps 是一组过程、方法与系统的统称，用于促进开发（Dev）、技术运营（Ops）和质量保障（QA）部门之间的沟通、协作与整合。



它是一种**文化思想**

打破部门墙。以前开发只管写代码，运维只管搬服务器，出了事互相甩锅。DevOps 思想要求**谁开发，谁运行**。它强调共享责任、免责复盘（**出了事查流程不查人**）和全员为最终价值负责。



它是一套**实践方法**

比如**CI/CD（持续集成/持续交付）**。这是一种规定的动作：规定代码必须每天提交、测试必须自动化、部署必须流水线化。它不指定你用什么软件，但指定了你必须按这个流程干活。



它是一组**工具链**

**DevOps 不是某个工具的名字。** 你不能说“我下载了一个 DevOps”。你只能说“我用 Jenkins 这个工具来实现 DevOps 的流程”。



### 闭环

DevOps 最经典的标志是一个无穷大符号（∞），代表了软件交付是一个**持续不断的闭环**。

![image-20260125134225712](./images/%E4%BA%91%E8%AE%A1%E7%AE%97.assets/image-20260125134225712.png)

这个环分为两个半区，但无缝连接：

**左半环（Dev - 开发侧）**

1. **Plan（计划）：** 定义需求，规划要开发的功能。
2. **Code（编码）：** 程序员编写代码。
3. **Build（构建）：** 将代码编译成可运行的软件包（Artifacts）。
4. **Test（测试）：** 自动化测试，确保质量。

**右半环（Ops - 运维侧）**

1. **Release（发布）：** 将通过测试的版本准备发布。
2. **Deploy（部署）：** 将软件部署到生产环境（如云服务器、K8s 集群）。
3. **Operate（运维/操作）：** 维护系统运行，管理配置。
4. **Monitor（监控）：** 实时监控系统状态，收集用户反馈。

监控（Monitor）发现的问题，会立即反馈到计划（Plan）阶段，开始下一轮的修复或优化。这就是**“持续反馈”**。



### C.A.L.M.S.

**核心支柱框架：C.A.L.M.S.**

行业内通常用 **C.A.L.M.S.** 模型来衡量一个团队是否实现了 DevOps：



**C - Culture (文化)**

这是最重要的。团队不再分“你们”和“我们”，大家有一个共同的目标：**为客户提供价值**。

- **免责文化 (Blameless)：** 出了事故（Bug），不是先查“是谁干的”并扣奖金，而是先查“流程哪里有漏洞”，如何防止下次再犯。



**A - Automation (自动化)**

手动操作是万恶之源（慢、容易出错）。DevOps 追求**“一切皆代码”**。

- 测试要自动化。
- 服务器配置要自动化（Infrastructure as Code）。
- 部署流程要自动化（CI/CD流水线）。



**L - Lean (精益)**

源自丰田生产方式。

- **小步快跑：** 不要憋三个月发一个大版本（风险极大），而是每天发三个小版本（风险极小）。
- **减少浪费：** 消除等待时间，消除重复劳动。



**M - Measurement (度量)**

你无法优化你无法度量的东西。DevOps 团队非常看重数据：

- **部署频率：** 一天发几次？
- **变更失败率：** 多少次发布导致了回滚？
- **MTTR (平均修复时间)：** 系统崩了，多久能修好？



**S - Sharing (共享)**

- **知识共享：** 开发懂一点运维知识，运维懂一点代码逻辑。
- **工具共享：** 开发和运维看同一个监控仪表盘，用同一套工单系统。



## CI/CD

**DevOps** 是指导思想，那么 **CI/CD** 就是把它落地的工具。

CI/CD 的全称是 **Continuous Integration（持续集成）** 和 **Continuous Delivery/Deployment（持续交付/持续部署）**。



**CI - 持续集成**

持续集成位于软件交付流水线的上游，其核心目标是降低代码集成的复杂度并实现缺陷的早期发现。	

在传统的开发模式中，由于代码合并频率较低，开发者长期在独立的分支上工作，导致主干代码与分支代码差异巨大。三个程序员合作开发。大家各自闷头写代码，写了一个月。到了月底，大家把代码合在一起（Merge）。结果发现张三改了底层逻辑，李四的代码全报错了。这种延迟合并极易引发严重的版本冲突，就是著名的**集成地狱**。

CI 实践要求开发人员以高频率（通常为每日多次）将代码提交至主代码仓库。每次提交均会自动触发构建系统，执行代码编译、静态分析以及自动化的单元测试。这种机制建立了一个快速反馈循环，一旦构建失败或测试未通过，系统会立即通知相关人员。

通过这种方式，团队能够确保主分支始终处于可构建、可测试的健康状态，从根本上杜绝了因长期未集成而积累的技术债务。



**持续交付与持续部署 (Continuous Delivery vs. Continuous Deployment)**

CD 涵盖了两个紧密相关但自动化程度不同的概念，二者的主要区别在于从测试环境到生产环境的发布过程是否需要人工干预。



**CD - 持续交付**

持续交付旨在确保软件随时处于可发布状态。在此模式下，代码通过 CI 阶段的自动化构建与测试后，会被自动部署到预发布环境或类生产环境。

将版本推向生产环境的最后一步仍保留了**人工审批**环节。这种模式在大多数企业中被广泛采用，因为它允许业务部门根据市场策略控制发布时机，或在合规性要求下进行最终的人工确认。



**CD - 持续部署**

持续部署是自动化流水线的终极形态。在此模式下，代码变更只要通过了流水线中所有的自动化测试与验证环节，便会自动部署至生产环境，全程**无需任何人工干预**。这种方式要求极高的自动化测试覆盖率和容错能力，通常由具备成熟 DevOps 能力的科技公司采用，以实现分钟级的高频发布与快速迭代。

一般只有 Facebook、Netflix、Google 等顶级互联网大厂会采用这种方式。他们的自动化测试极其严密，有信心不靠人眼检查。



## GitOps

GitOps 是 CI/CD 在云原生时代的**最佳实践**，它**本质上是一种进行 CD  的新方法**。

- 传统的 CD 往往是 CI 服务器直接把代码推（Push）到服务器。
  - 程序员提交代码 -> CI 工具（Jenkins/GitLab）运行测试 -> 构建镜像 -> **CD** 工具持有集群密钥，执行命令（如 `kubectl apply`）将应用推送到生产环境。
- GitOps 改变了 CD 的方式，变成了服务器自己从 Git 仓库拉（Pull）配置。
  - 程序员提交代码 -> CI 工具运行测试 -> 构建镜像 -> **CI 工具只更新 Git 仓库里的配置文件（YAML）** -> **集群内部的 Agent（如 ArgoCD）发现 Git 变了，自己把变更拉下来并应用**。



**GitOps 既可以是持续交付，也可以是持续部署，这完全取决于你怎么设计 Git 仓库变更流程这一步。**

GitOps 的本质是**自动化同步**（只要 Git 变了，集群就跟着变）。但是，Git 仓库变更流程，决定了它到底是交付还是部署。



**作为持续部署**

这是最理想化的 GitOps 流程，全程无人值守。

**CI 机器人自动**修改配置仓库里的 YAML 文件（把 `v1.0` 改为 `v2.0`），并**自动提交（Commit & Push）**到主分支。

GitOps 控制器（如 ArgoCD）发现主分支变了，立即自动将集群更新到 `v2.0`。

从提交代码到用户用上新版本，全程无人工干预。这就是**持续部署**。



**作为持续交付**

CI 机器人（或者开发者）**不会直接修改主分支**，而是创建一个 **Pull Request (PR/MR)**，提议将配置仓库的版本从 `v1.0` 改为 `v2.0`。

团队负责人或 QA 收到通知，查看这个 PR。确认无误后，**人工点击合并 (Merge)按钮**。

PR 一旦合并，GitOps 控制器发现主分支变了，这才开始同步集群。

流程中存在一个**人工审批（点击 Merge）\**的动作。这就是\**持续交付**。
