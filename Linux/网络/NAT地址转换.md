# NAT 地址转换

网络地址转换（Network Address Translation，简称 NAT）是一种网络层技术，其核心功能是在 IP 数据包通过网络边缘设备（如路由器或防火墙）时，**对数据包报头中的源 IP 地址或目的 IP 地址进行重写**。

该技术的产生初衷是为了缓解 IPv4 地址枯竭的压力，通过允许私有网络内的**多台主机共享少量的公网 IPv4 地址来访问互联网**。

在 TCP/IP 模型中，**NAT 主要基于网络层信息进行操作**，但当涉及到端口转换（NAPT）时，它也会读取并修改传输层的 TCP 或 UDP 端口信息。NAT 设备在转发数据包时，不仅会修改源 IP 地址或目的 IP 地址，还必须重新计算 IP 报头的校验和，以保证数据包的完整性。

NAT 设备会在内存中维护一张动态或静态的会话表（Session Table），精确记录内网 IP、端口与外网 IP、端口之间的映射关系，确保后续的回程数据包能够准确无误地被逆向转换并转发回内网的发起主机。



## NAT 类型

### 静态 NAT

**静态网络地址转换 (Static NAT)**

静态网络地址转换采用的是一种固定且永久的一对一地址映射机制。

在配置过程中，**网络管理员需要在 NAT 设备上手动建立内网私有 IP 地址与外网公有 IP 地址之间的对应条目。**这种映射关系一旦确立，便永久生效，不受数据流量是否存在或内网主机是否在线的影响。

在静态 NAT 环境下，当特定的内网主机向外发送数据时，NAT 设备始终将其源 IP 地址转换为预先指定的同一个公网 IP 地址；反之，当外部网络访问该公网 IP 时，目的地址也会被恒定地转换为对应的内网私有 IP。

尽管静态 NAT 无法实现公网 IP 地址资源的节省（因为每个内网主机都需要独占一个公网 IP），但它具备极其稳定的双向通信能力，外部网络可以随时主动发起对内网主机的访问。

这种类型通常被应用于需要对外提供持续服务的内部服务器，如 Web 服务器、FTP 服务器或企业邮件服务器。



### 动态 NAT

**动态网络地址转换 (Dynamic NAT)**

动态网络地址转换虽然也遵循一对一的转换原则，但其映射关系的建立是临时的、动态分配的。

管理员预先在 NAT 设备上定义一个公网 IP 地址池。

当内网主机发起对外连接请求时，NAT 设备会检查地址池，从中选取一个当前未被占用的公网 IP 地址，将其与发起请求的内网私有 IP 地址进行绑定，并在 NAT 表中生成临时映射条目。

一旦通信会话结束，或者在设定的时间内没有数据流量传输导致超时，NAT 设备会删除该映射条目，并将对应的公网 IP 地址回收至地址池中，供其他内网主机使用。

动态 NAT 相比静态 NAT 提高了一定的地址利用率，但由于在同一时刻一个公网 IP 只能服务于一台内网主机，因此其并发连接数严格受限于地址池中公网 IP 的数量。如果地址池耗尽，后续的内网主机将无法建立连接，直到有 IP 资源被释放。



### 端口转换 NAT

**网络地址端口转换 (NAPT/PAT)**

网络地址端口转换（Network Address Port Translation），常被称为端口地址转换（PAT），是目前解决 IPv4 地址短缺最为有效的方案。

NAPT 打破了一对一的限制，允许多个内网主机通过共享同一个公网 IP 地址来访问互联网。

它通过利用传输层的端口号（Port）作为区分不同会话的标识符。当内网主机发起连接时，NAT 设备不仅将源 IP 地址转换为公网 IP，还会将源端口号替换为一个在公网 IP 上未被占用的临时端口号。

NAT 设备在转换表中记录的是“内网 IP + 内网端口”与“公网 IP + 公网端口”的映射关系。当回程数据包到达时，设备依据目的端口号查询映射表，将数据包准确转发给对应的内网主机。

NAPT 使得一个公网 IP 理论上可以支持多达 65535 个并发会话（受限于端口总数），极大地提升了公网地址的复用率。



## NAPT 细分

RFC 3489 是一个非常经典的网络协议标准，它的全称是 **STUN (Simple Traversal of User Datagram Protocol (UDP) Through Network Address Translators (NATs))**，即“UDP 穿透 NAT 的简单方法”。

虽然该协议目前已经被 **RFC 5389** 取代（名称变更为 Session Traversal Utilities for NAT），但 RFC 3489 定义的 **NAT 类型分类**至今仍被广泛引用，是理解网络穿透（P2P、VoIP、WebRTC）的基石。

RFC 3489 协议中定义了 NAT 四种类型（全锥型、地址限制锥型、端口限制锥型、对称型）。



NAPT（网络地址端口转换）与全锥型、地址限制锥型、端口限制锥型及对称型 NAT 之间的关系。

NAPT 是底层实现机制，即通过修改传输层端口号来实现多台内网主机共享公网 IP 的技术手段；而上述四种 NAT 类型则是基于 NAPT 机制之上，根据设备对“映射关系的持久性”以及“对外部流量的过滤策略”所展现出的具体行为模式。

无论是家庭路由器还是运营商边缘节点，它们在执行地址转换时本质上都在使用 NAPT 技术，但根据其配置策略和防火墙规则的不同，会表现出特定的 NAT 行为类型。



技术论坛中常说的 NAT1 - NAT4 通常指的是：

NAT1：全锥型 NAT

NAT2：地址限制锥型 NAT

NAT3：端口限制锥型 NAT

NAT4：对称型 NAT



###  NAT

**全锥型 NAT (Full Cone NAT)**

**全锥型 NAT** 是所有**锥型 NAT** 中限制最为宽松的一种类型，其核心特征在于映射关系的唯一性和对外部流量的完全开放性。

在映射规则上，一旦内网主机（内网 IP:内网端口）发起向外的连接，NAT 设备会为其分配一个固定的公网映射（公网 IP:公网端口）。在此后的任何时间，只要该会话未超时，无论该内网主机是向目标 A 发送数据，还是向目标 B 发送数据，其使用的公网映射（公网 IP:公网端口）始终保持不变。

在映射规则上，一旦内网主机（内网 IP:内网端口）发起向外的连接，NAT 设备会为其分配一个固定的公网映射（公网 IP:公网端口）。在此后的任何时间，只要该会话未超时，无论该内网主机是向目标 A 发送数据，还是向目标 B 发送数据，其使用的公网映射（公网 IP:公网端口）始终保持不变。

这种特性使得全锥型 NAT 对于 P2P 通信极其友好，因为它允许任何外部节点主动连接内网主机。



### 地址限制锥型 NAT

**地址限制锥型 NAT (Restricted Cone NAT)**

**地址限制锥型 NAT**，在映射机制上与全锥型 NAT 保持完全一致，但在接收外部流量时增加了基于源 IP 地址的过滤条件。

在映射规则方面，当内网主机（内网 IP:内网端口）通过 NAT 设备向外发送数据时，设备会为其分配一个固定的公网映射（公网 IP:公网端口）。无论该内网主机后续向哪个外部目标发送数据，只要其源 IP 和源端口不变，NAT 设备始终使用这同一个公网 IP 和公网端口进行映射，这一点与全锥型 NAT 完全相同。

在过滤规则上，地址限制锥型 NAT 实施了安全检查：只有当内网主机曾经主动向某个外部 IP 地址发送过数据包时，NAT 设备才会接收来自该外部 IP 地址的数据包并转发回内网。

如果内网主机从未向 IP 地址为 X 的外部主机发送过数据，那么即便 X 知道了映射后的公网端口，其发送的数据包也会被 NAT 设备直接丢弃。此类型只验证外部发送者的 IP 地址，而不验证其端口号。



### 端口限制锥型 NAT

**端口限制锥型 NAT (Port Restricted Cone NAT)**

**端口限制锥型 NAT** 是在**地址限制锥型 NAT** 的基础上，进一步将过滤规则细化到了传输层的端口级别。

在映射规则方面，其行为依然与**全锥型 NAT** 和**地址限制锥型 NAT** 保持高度一致：内网主机（内网 IP:内网端口）在向外发起连接时，会被映射到一个固定的公网映射（公网 IP:公网端口）。

无论该内网主机是向外部目标 A 还是外部目标 B 发送数据，只要内网源地址和端口未变，其在公网侧呈现的 IP 和端口就始终保持不变，此处与前述两种锥型 NAT 一样。

在过滤规则上，**端口限制锥型 NAT** 极为严格：外部主机要想成功发送数据给内网主机，不仅其源 IP 地址必须是内网主机曾经访问过的，其源端口号也必须是内网主机曾经访问过的。

只有当内网主机向外部主机 X 的端口 P 发送过数据后，主机 X 才能通过端口 P 向内网主机回发数据。任何来自不同 IP 或同一 IP 但不同端口的数据包，都会被 NAT 设备阻断。

这种机制显著提高了安全性，但也极大地增加了 P2P 打洞的难度。



### 对称型 NAT

**对称型 NAT (Symmetric NAT)**

对称型 NAT 是所有 NAT 类型中行为最特殊、限制最严格的一种，其映射机制与锥型 NAT 存在本质区别。在对称型 NAT 中，映射关系不再仅仅取决于内网源地址，而是与“源地址 + 目的地址”这一组合紧密绑定。

当内网主机使用相同的（内网 IP:内网端口）向外部目标 A 发送数据时，NAT 设备会为其分配一个公网映射（公网 IP 1:公网端口 1）；但当该内网主机使用完全相同的（内网 IP:内网端口）向另一个外部目标 B 发送数据时，NAT 设备会识别出目的地址的变化，从而为其分配一个新的、不同的公网映射（公网 IP 1:公网端口 2）。外部观察者无法根据与内网主机的一个连接来推测其与其他连接的端口映射关系。

在过滤规则上，对称型 NAT 同样执行严格的检查，仅允许特定的外部目标（特定的 IP 和端口）在收到数据后进行回发。由于映射端口随目标变化，传统的 STUN 等打洞技术在对称型 NAT 下几乎完全失效，通常必须依赖 TURN 服务器进行流量中继。



## 运营商级 NAT

**运营商级 NAT (CGNAT)**

运营商级 NAT（Carrier-Grade NAT），也被称为大规模 NAT（LSN），是部署在互联网服务提供商（ISP）骨干网中的一种高容量地址转换技术。

由于公网 IPv4 地址的极度匮乏，ISP 无法为每个宽带用户分配独立的公网 IPv4 地址。在 CGNAT 架构下，用户的家庭网关获取到的 WAN 口地址并非公网 IP，而是一个位于运营商内网的私有 IP（通常使用 IANA 预留的 100.64.0.0/10 地址段）。

当用户流量流向互联网时，数据包首先在用户家庭路由器进行一次 NAT 转换，随后在运营商边缘节点进行第二次 NAT 转换，最终才以公网 IP 进入互联网。

这种“多重 NAT”架构导致用户无法直接从外网访问家庭内部网络，常规的端口映射功能失效，同时也给网络应用的端到端连接和溯源带来了极大的复杂性。



**家庭路由器中的 NAPT 类型归属：端口限制锥型 NAT**

绝大多数默认配置下的现代家庭宽带路由器（Home Router），其 NAPT 实现行为属于**端口限制锥型 NAT (Port Restricted Cone NAT)**。



**运营商级 NAT (CGNAT) 中的 NAPT 类型归属：端口限制锥型 NAT 或对称型 NAT**

在运营商边缘节点部署的 CGNAT 环境中，其 NAPT 行为通常表现为**端口限制锥型 NAT或对称型 NAT (Symmetric NAT)**。

不同运营商的 CGNAT 中的 NAPT 类型是不同的，同一运营商不同地域（通常指的是不同省）的 NAPT 类型也是不同的

在有一些省对 PCDN 的打击力度比较大，家庭用户宽带的 NAPT 类型通常是对称型 NAT。

有一些省，对 PCDN 基本不做管控，家庭用户宽带的 NAPT 类型是端口限制锥型 NAT。

**注意：部分地区的运营商也可能是地址限制型 NAT**



**企业级路由器中的 NAPT 类型归属：对称型 NAT**

在企业级网络架构中，部署在互联网出口边界的专业路由器或下一代防火墙（NGFW），当配置了基于 NAPT 的源地址转换（通常称为 Dynamic PAT 或 Source NAT）时，行业公认的默认行为标准是 **对称型 NAT (Symmetric NAT)**。

与家庭路由器为了兼顾游戏和语音通信体验而默认采用端口受限圆锥型 NAT 不同，企业级设备的设计哲学优先考虑网络安全性与会话的隔离性。

在这种模式下，防火墙在处理出站流量时，会严格基于“五元组”（源 IP、源端口、目的 IP、目的端口、协议）来分配公网端口。

这意味着，即便是同一台内网主机的同一个源端口，在连续访问互联网上两个不同的目的服务器时，企业级防火墙也会强制为其分配两个完全不同的公网端口。

这种机制在最大程度上隐藏了内网主机的流量特征，使得外部攻击者无法通过猜测端口规律来进行会话劫持或网络拓扑探测。



尽管对称型 NAT 是默认标准，但在企业网络管理中，这种行为是完全可以修改和定制的，以适应特定的业务需求。



**对于需要被外部稳定访问的内部服务器，管理员通常会绕过 NAPT，直接配置静态 NAT（Static NAT / 1:1 NAT），从而彻底消除端口动态转换带来的连接障碍。**



## P2P 打洞

要判断哪两种 NAT 类型结合可以实现 P2P 连接，首先必须理解 P2P 通信中核心的“UDP 打洞”（Hole Punching）机制。



该技术的成功依赖于两个关键因素：

- 一是 NAT 设备能否保持映射关系的一致性，即内网主机向不同目标发送数据时是否使用相同的公网端口；
- 二是 NAT 设备的防火墙过滤规则是否足够宽松，允许外部数据包通过已建立的映射进入内网。



在理想的 P2P 连接建立过程中，通信双方会预先通过一个公网的中介服务器（如 STUN 服务器）交换各自的公网 IP 和端口信息，然后同时或按序向对方发送数据包。

如果双方的 NAT 类型允许这种直接的交互，连接即告建立；反之，则连接失败，流量必须通过中继服务器（TURN）转发。



### 三种锥型 NAT

在锥型 NAT 的三大类别——全锥型、地址限制锥型和端口限制锥型之中，任意两种类型之间的组合均能成功建立 P2P 连接。

这是因为锥型 NAT 的核心特性是“端点独立映射”，即只要内网源端口不变，公网映射端口就不变。当双方都是锥型 NAT 时，它们通过 STUN 服务器获取的公网地址对于直接通信也是有效的。其中，全锥型 NAT 由于没有任何入站限制，可以与任何类型的 NAT（包括对称型）瞬间建立连接。

对于地址限制锥型和端口限制锥型的组合（例如“端口限制 + 端口限制”），虽然双方最初发出的第一个数据包都会被对方的防火墙拦截（因为此时对方尚未建立对应的出站记录），但这个发送动作本身会在己方的 NAT 设备上“打”开一个洞（建立会话条目）。

当双方都完成了发送动作后，后续的数据包就能匹配到对方已存在的会话记录，从而穿透防火墙实现互通。这也是绝大多数现代网络游戏和 P2P 软件在家庭宽带环境下能够正常工作的基本场景。



### 对称型 NAT 与锥型 NAT

当通信的一方涉及到对称型 NAT 时，情况变得复杂，其连通性取决于另一方 NAT 对入站流量的限制严格程度。

对称型 NAT 可以与全锥型 NAT 成功连接，因为全锥型不检查外部数据包的源 IP 和源端口，无论对称型 NAT 一侧如何改变公网映射端口，全锥型一侧都会照单全收。

对称型 NAT 也可以与地址限制锥型 NAT（仅 IP 受限）成功连接。这是因为虽然对称型 NAT 会改变公网端口，但其公网 IP 通常保持不变（除非是多 WAN 口环境），而地址限制锥型 NAT 仅校验源 IP 地址，忽略端口变化，因此数据包依然可以通过验证。



### 对称型 NAT

在涉及对称型 NAT 的组合中，存在两种无法进行直接 P2P 连接的“死局”。



 **对称型 NAT 与端口限制锥型 NAT 的组合**

这是互联网中极为常见的一种失败案例，因为家庭网络大多是端口限制型，而移动网络或企业网常是对称型。在这种场景下，对称型 NAT 一侧在向端口限制型一侧发送数据时，会分配一个新的、不可预测的公网端口。

端口限制锥型 NAT 严格要求入站数据包的源端口必须与内网主机先前访问过的端口一致（即 STUN 服务器看到的那个端口）。由于对称型 NAT 改变了端口，导致数据包无法匹配端口限制侧的过滤规则，从而被丢弃。



**对称型 NAT 与对称型 NAT 的组合**

在这种情况下，双方都会因为目标地址的变化而改变各自的公网映射端口，且双方都无法预知对方的新端口，导致无法建立有效的握手，连接必然失败。在这两种阻断场景下，通信必须回退到使用 TURN 服务器进行中继转发的模式。



## DMZ

DMZ（Demilitarized Zone，隔离区）在网络安全架构中指的是一个位于内部受信网络（Internal Network）与外部不可信网络（Internet）之间的物理或逻辑子网。

其核心设计目的是建立一个安全缓冲区，将对外提供服务的服务器（如 Web 服务器、邮件服务器、DNS 服务器等）部署在 DMZ 区中。



这种架构实施了严格的访问控制策略。通常情况下，外部网络被允许访问 DMZ 区中的特定服务，DMZ 区内的服务器被允许响应外部请求，但严格禁止 DMZ 区主动发起连接去访问内部核心网络。

通过这种方式，即使部署在 DMZ 区的服务器遭到攻击或沦陷，攻击者也无法直接以此为跳板无限制地渗透到存有敏感数据的内部局域网，从而将安全风险控制在有限的范围内。



### 家庭路由器 DMZ 功能

在普通家用或小型办公（SOHO）路由器中，DMZ 的概念被简化为“DMZ 主机”。给路由器设置 DMZ IP，实质上是配置了一种全端口映射规则，或者称为默认的 NAT（网络地址转换）转发策略。

当启用该功能并指定内部局域网的某一台主机的 IP 地址（即 DMZ IP）后，路由器会将所有从广域网（WAN）接口进入、且没有匹配到具体“虚拟服务器”或“端口转发”规则的数据包，全部无条件地转发给这台指定的内部主机。这相当于将该内部主机完全暴露在公网环境中，绕过了路由器防火墙对入站流量的默认阻挡机制。



将特定 IP 设置为 DMZ 主机后，路由器会转发**所有基于 IP 协议（Internet Protocol）传输的有效载荷数据**，除非该数据包的目标端口已被其他更高优先级的规则占用。具体包含的协议类型如下：

- **TCP（传输控制协议）数据包**，覆盖从 0 到 65535 的所有端口号，这确保了 HTTP、HTTPS、FTP、SSH 等依赖 TCP 的应用流量能通过。
- **UDP（用户数据报协议）数据包**，同样覆盖所有端口，这支持了 DNS 查询、流媒体传输、在线游戏以及部分 VoIP 通信。
- **ICMP（互联网控制消息协议）**、**GRE（通用路由封装协议）**、**ESP（封装安全载荷协议，IPSec 组件）** 以及 **AH（认证头协议，IPSec 组件）**等数据包。

从技术严谨性角度来看，路由器是依据 IP 数据包报头中的 **Protocol 字段**进行判断的。只要 NAT 引擎支持，任何符合 IPv4 标准协议号定义的流量（包括但不限于 SCTP 等较少见的协议），只要没有明确的规则将其分流到其他主机，都会被默认发送至 DMZ 主机。



## EIP

在阿里云等公有云环境中，ECS 云服务器绑定 EIP（弹性公网 IP）却在本地网卡只能看到私网 IP，这是**1:1 NAT（静态网络地址转换）**技术在云网络边界的具体应用。

EIP 实际上是存在于云服务商的公网网关（Internet Gateway）或虚拟交换机层面的逻辑地址，并不直接配置在 ECS 实例的虚拟网卡（vNIC）上。当公网数据包发送给 EIP 时，数据包首先到达云服务商的网关设备。

网关设备通过查找路由表和 NAT 映射表，将数据包的目标 IP 地址从 EIP 修改为 ECS 实例对应的私网 IP 地址（即 DNAT，目标地址转换），然后再将修改后的数据包投递到 ECS 实例的网卡。

当 ECS 实例向公网发送数据时，数据包的源 IP 是其私网 IP。这些数据包在流出云网络边界之前，会被网关设备截获，并将源 IP 地址修改为绑定的 EIP（即 SNAT，源地址转换），最后发送到互联网。

这种架构实现了公网地址与云主机操作系统的解耦，使得 EIP 可以灵活地在不同实例间解绑和绑定，同时简化了操作系统层面的网络配置，因为操作系统只需要维护通过 DHCP 获取的私网 IP 配置即可，无需感知公网 IP 的变化。



### 1:1 NA

在启用 1:1 NAT 的场景下，系统会转发**基于 IPv4 协议栈构建的所有有效载荷数据包**。这不仅包含最常见的TCP（传输控制协议）和UDP（用户数据报协议）的所有端口流量，也毫无遗漏地包含ICMP（互联网控制消息协议）数据包，从而支持 Ping 探测和 MTU 路径发现。

**简而言之，只要 IP 报头中的 Protocol 字段值在 IANA 定义的 0 到 255 之间，且数据包符合 IP 校验和标准，该数据包就会被转发。**



### Linux 模拟 1:1 NAT

在标准的 Linux 系统中，要实现与云环境完全一致的 1:1 NAT 效果，需要利用内核自带的 Netfilter 框架，并通过用户空间工具 `iptables` 进行规则配置。这需要同时配置 DNAT（目的地址转换）和 SNAT（源地址转换）来实现双向流量的对应。



假设 Linux 网关拥有公网 IP（例如 1.1.1.1）配置在 eth0 接口，内部私网主机 IP 为 192.168.1.100。



开启 Linux 内核的 IP 转发功能，这是实现路由和 NAT 的基础。这通常通过修改 `/etc/sysctl.conf` 文件中的 `net.ipv4.ip_forward` 参数为 1 并加载生效来实现。如果该开关未打开，Linux 将直接丢弃非本机目的地的流量，后续的 NAT 规则将无法生效。



针对从公网进入私网的入站流量，需要在 `nat` 表的 `PREROUTING` 链中添加 DNAT 规则。该规则的逻辑是：当内核检测到进入网卡的数据包目的地址为公网 IP（1.1.1.1）时，在路由决策发生之前，将目的地址修改为内部主机的私网 IP（192.168.1.100）。具体的命令逻辑如下：

```
iptables −t nat −A PREROUTING −d 1.1.1.1 −j DNAT −−to−destination 192.168.1.100
```



针对从内部主机发往公网的出站流量，需要在 `nat` 表的 `POSTROUTING` 链中添加 SNAT 规则。该规则的逻辑是：当数据包经路由判断即将离开网卡时，如果源地址是内部主机的私网 IP（192.168.1.100），则将源地址修改为公网 IP（1.1.1.1）。这一步至关重要，它确保了回程数据包能够正确返回。具体的命令逻辑如下：

```
iptables −t nat −A POSTROUTING −s 192.168.1.100 −j SNAT −−to−source 1.1.1.1
```



在标准 Linux 系统中，要彻底避免本地应用（如 SSH、Nginx 或系统自动更新进程）绑定或使用用于 1:1 NAT 的公网 IP 地址，最根本的解决方案是**不在网络接口上配置该 IP 地址**。

通常情况下，我们习惯使用 `ip addr add` 将 IP 配置在网卡上，这会产生一条“本地路由（Local Route）”，告知内核该 IP 归本机所有，从而导致本机应用可以监听该 IP 上的端口，或在向外发起连接时被系统自动选择为源 IP。

为了实现类似公有云 EIP 的效果，即该 IP 对宿主机操作系统“不可见”但又能被网关逻辑转发，需要结合 **Proxy ARP（代理 ARP）** 技术与 **Netfilter** 框架来实现。此时，Linux 主机仅仅扮演“流量黑洞”和“转发器”的角色，而不是该 IP 的终端所有者。



由于没有将公网 IP 配置在网卡上，上级网关（路由器）在尝试发送数据时无法通过 ARP 协议获取该 IP 对应的 MAC 地址。

必须在 Linux 网关的 WAN 口开启 Proxy ARP 功能，并手动添加一条邻居表项。这使得 Linux 内核能够代替那个并不真正属于本机的公网 IP 响应 ARP 请求。

确保内核参数 `net.ipv4.conf.eth0.proxy_arp` 已启用（假设 eth0 为公网接口）。

使用 `ip neigh` 命令添加代理规则，例如：`ip neigh add proxy 1.1.1.1 dev eth0`。

此时，流量能够到达网卡，但操作系统网络栈不会将其视为本地流量，普通应用也就无法调用该 IP。



当数据包到达网卡后，由于目的地址不是本机配置的任何 IP，标准行为是将其丢弃或尝试转发。

此时需要利用 `iptables` 在路由决策前的 `PREROUTING` 阶段介入。 配置与之前相同的 DNAT 规则：`iptables -t nat -A PREROUTING -d 1.1.1.1 -j DNAT --to-destination 192.168.1.100`。 

Netfilter 的处理优先级高于本地路由表查找。当数据包被 DNAT 规则捕获并修改目的地址为 192.168.1.100 后，内核随后的路由决策会将其判定为需要转发（Forward）给内网的数据包，而非提交给本地层（Local Input）。



对于出站流量，继续配置 SNAT 规则：`iptables -t nat -A POSTROUTING -s 192.168.1.100 -j SNAT --to-source 1.1.1.1`。



通过这种“无地址接口”的配置方式，该公网 IP 1.1.1.1 对于运行在 Linux 网关上的任何应用程序都是不可见的，没有任何 socket 可以绑定它，系统也绝不会将其自动选为出站源地址，从而完美实现了与业务流量的彻底隔离。
