# 存储

## 硬盘

硬盘是用来存储数字信息的设备。数据最终以二进制形式（0和1）存储在物理介质上。硬盘属于非易失性存储，意味着即使断电，数据也不会丢失。

两种主要类型为**机械硬盘**和**固态硬盘**。



### 机械硬盘

**机械硬盘 (HDD - Hard Disk Drive)**

HDD通过高速旋转的**盘片**（Platter）和在其上方移动的**读写磁头**（Read/Write Head）来读写数据。盘片表面涂有磁性材料，数据以磁化的方式记录在盘片上的同心圆**磁道**（Track）和扇区（Sector）中。

机械部件的存在使得HDD在读写速度上受限，容易受到震动影响，但具有成本低、容量大的优势。

![机械硬盘插图1](./images/%E5%AD%98%E5%82%A8%E5%9F%BA%E7%A1%80.assets/unnamed-file-441.jpg)



#### 盘片

盘片是硬盘的存储介质。是一个表面极其光滑、涂有磁性材料的圆形金属或玻璃盘。

一块硬盘内可以有一张或多张盘片，它们重叠安装在同一个**主轴 (Spindle)** 上，同步高速旋转（如 7200 RPM）。

每个盘片有两个面（上表面和下表面）。通常每个面都可以用来存储数据（但在某些大容量硬盘中，可能只使用其中一个面，或者为了良品率屏蔽掉一个面）。



#### 磁头

磁头负责读写数据的组件。

**每个有效的盘面都配备一个磁头**。磁头从 0 开始编号。如果你有 2 张盘片，且双面都使用，那么就有 4 个磁头（Head 0, 1, 2, 3）。

磁头悬浮在高速旋转的盘面上方（距离仅几纳米），像唱针一样，但**绝对不接触**盘面。



#### 磁道

磁道盘片表面上划分出的无数个同心圆环。

数据是沿着这些圆环（磁道）被记录下来的。从盘片的最外圈开始向内圈编号，最外圈是 0 磁道。



#### 柱面 

柱面是一个**逻辑上的立体概念**，也是理解机械硬盘最关键的术语。

柱面数量也就是也就是磁道（Track）数。所有盘面上半径相同的磁道组成的圆柱面。

所有盘面上、半径相同的磁道，在垂直方向上重叠构成的圆柱面。盘片 A 上面的 100 号磁道，盘片 A 下面的 100 号磁道，盘片 B 上面的 100 号磁道... 它们共同组成了“第 100 号柱面”。

机械硬盘读写时，所有磁头是**同步移动**的。当磁头臂定位到“第 100 号柱面”时，所有磁头都同时停在各自盘面的第 100 号磁道上。**系统读写数据时，倾向于先写满一个柱面（因为不需要移动磁头臂），再去下一个柱面，这样速度最快。**



#### 扇区

扇区是磁道被切割成的更小的弧段，是硬盘读写数据的**最小物理单位**。无论你要修改哪怕 1 个字节的数据，硬盘也必须把整个扇区（512B 或 4KB）读出来，修改后再写回去。

- **传统标准：** 512 字节 (Bytes)。
- **现代标准 (4K Native / Advanced Format)：** 4096 字节 (4KB)。这是为了提高纠错效率和存储密度。





#### 容量计算



##### CHS 公式

计算硬盘容量主要通过三个核心参数：**柱面数 (C)**、**磁头数 (H)**、**扇区数 (S)**。这被称为 **CHS 寻址模式**。

**传统计算公式 (CHS 公式)**
$$
硬盘总容量=柱面数 (Cylinders)×磁头数 (Heads)×每磁道扇区数 (Sectors)×扇区大小 (通常 512B)
$$

**举个老式硬盘的例子：**

- 柱面数：1024
- 磁头数：16
- 每磁道扇区数：63
- 扇区大小：512 字节
- **计算：** 1024×16×63×512≈528 MB



##### 现代硬盘计算公式

**现代硬盘的实际计算 (ZBR 技术与 LBA)**

上述公式有一个巨大的物理缺陷：**内圈和外圈的周长不同**。如果外圈和内圈的扇区数一样多（如老式硬盘），外圈的数据密度就会很稀疏，浪费空间。

现代硬盘使用了 **ZBR (Zone Bit Recording，区域位记录)** 技术：

- **外圈磁道长：** 划分更多的扇区。
- **内圈磁道短：** 划分较少的扇区。

因此，现代硬盘**没有固定的“每磁道扇区数”**，CHS 公式在物理上已经失效了。为了兼容电脑系统，硬盘内部的主控芯片会把复杂的物理结构“翻译”成一个线性的地址，这就是 **LBA (Logical Block Addressing，逻辑块寻址)**。



**现代硬盘容量计算方法：** 查看硬盘标签上的 **LBA 总数**（Total LBA Sectors）。
$$
硬盘总容量=LBA 总数×扇区大小 (512B 或 4KB)
$$



### 固态硬盘

**固态硬盘 (SSD - Solid State Drive)**

SSD使用**NAND闪存**（Flash Memory）芯片来存储数据，不含任何机械部件。数据以电荷的形式存储在存储单元中。

无移动部件，因此具有高速读写、抗震动、低功耗、无噪音的优势，但成本相对较高，相同容量下价格通常高于HDD。



## 基础知识

### 存储单位

在计算机和通信领域，区分它们的核心在于两个维度：

1. **大小写之分：** `B` (Byte, 字节) 与 `b` (bit, 位/比特)。
2. **进制之分：** `G` (Giga, 十进制 1000) 与 `Gi` (Gibi, 二进制 1024)。



#### GB

GB (Gigabyte) - 吉字节

**全称** Gigabyte，是**十进制**存储单位，由 **SI（国际单位制）**标准定义。
$$
1 GB=1000MB=10^9Bytes=1,000,000,000 Bytes。
$$



#### GiB

GiB (Gibibyte) - 吉比字节

GiB  是 Gibibyte (Giga **binary** byte 的缩写)，它是**二进制**存储单位。

由**IEC（国际电工委员会）标准 60027-2** 定义。为了解决“1000还是1024”的歧义，于 1998 年正式制定。
$$
1 GiB = 1024MiB = 2^{30}Bytes = 1,073,741,824Bytes
$$



#### Gb

Gb (Gigabit) - 吉比特

Gb 是 Gigabit（注意这里是 bit，不是 byte）的缩写，别名 **Gbit**。

表示信息的**数量**，基于**十进制**，但是单位是“位”。
$$
1 Gb = 1000Mb = 10^9 bits=1,000,000,000 bits
$$

$$
1 Byte=8 bits，1 GB=8 Gb
$$



#### Gbps

Gbps (Gigabits per second) - 吉比特每秒

Gbps 是 Gigabits per second 的缩写，别名 **Gb/s**，主要用于描述数据**传输速率**（带宽）单位。

运营商宣传的“千兆宽带”，指的是 1000 Mbps 或 1 Gbps。1 Gbps 的理论下载速度是 125 MB/s（1000 / 8）。
$$
1Gbps = 1000 Mbps = 1,000,000 Kbps
$$



#### GBps

GBps (Gigabytes per second) - 吉字节每秒

GBps 是 Gigabytes per second 的简写，别名有  **GB/s**（注意 **B** 是大写的），使用十进制表示，每秒传输的**字节**数。

**GBps** 理论上应该遵循 SI 标准（十进制），但在**某些软件或操作系统报告的实际传输速率**中，它可能被非正式地当作 **GiBps** 来使用。

主要用于描述本地读写速度（例如 SSD 硬盘、内存带宽的速度）。例如 PCIe 4.0 SSD 的速度可达 7 GB/s。
$$
1 GBps = 1000KBps = 1,000,000 Bps
$$



#### Gib

Gib (Gibibit) - 吉比比特

Gib 是 **二进制**的 bit 单位。1GiB = 8 Gib，1Gib = 1/8 GiB。

极少在消费者领域出现，主要用于极其精确的底层硬件架构文档或特定的映射协议中，用来描述芯片级的存储密度。
$$
1 Gib= 1024 Mib = 2^{30} bits=1,073,741,824 bits。
$$

$$
1 Gib = 1024 Mib = 128 MiB
$$



**Linux:** 现在的版本显示 "GiB"，计算用的也是 GiB (1024 进制)。

**macOS:** 现在的版本显示 "GB"，计算用的也是 GB (1000 进制)。所以在 Mac 上，1TB 硬盘通常就显示为 1TB，与厂商标称一致。

**Windows:** 显示的是 "GB"，但实际计算用的是 GiB (1024 进制)，**它错误地将单位显示为 GB**（这是混淆的万恶之源）。



## 性能指标

### IOPS

**IOPS (Input/Output Operations Per Second，每秒输出/输出操作次数)** 是存储性能中最常被引用，但也最容易被误读的指标。

从专业和架构的深度来看，IOPS 并非一个孤立的静态数值，而是一个**多维度的动态函数**。它描述的是存储系统在单位时间内处理读写请求的能力，本质上反映了系统的**事务处理能力**和**并发能力**。

在没有任何限定条件的情况下谈论 IOPS 是没有意义的，一个有效的 IOPS 数值必须包含**块大小、访问模式和队列深度**三个核心变量（即测试基准）。



#### 块大小

**块大小 (Block Size / Chunk Size)**

硬盘的**扇区 (Sector)**是硬盘（硬件）读写的最小物理单位。

- 机械硬盘（HDD）通常是 **512 Bytes**（老式）或 **4KB**（现代，称为高级格式化 4Kn）。
- 固态硬盘（SSD）内部读写单位其实是 Page（通常 **4KB - 16KB**），但在对外接口上通常伪装成 512B 或 4KB 扇区。



单个请求中的 **I/O 请求块大小**的**最小块大小**是**512 Bytes** 或 **4 KB** (取决于硬盘的物理扇区大小)。



**任何硬盘接口和介质都有一个“最大吞吐量（带宽）”的物理上限**，要理解**大小越大，IOPS 越小**这个问题，需要引入一个存储性能的“黄金公式”，并从物理传输耗时的角度来剖析。

存储性能之间存在一个铁定的数学关系，带宽守恒定律 (黄金公式)：
$$
带宽(MB/s)=IOPS×块大小(BlockSize)
$$
把这个公式变换一下，用来计算 IOPS：
$$
IOPS= \frac{块大小(BlockSize)}{带宽(MB/s)}
$$


假设 SSD 连续读写带宽极限是 **5000 MB/s**（这是物理上限，类似于水管的最粗直径，无法突破）。

**当传输小块数据 (4KB) 时：**
$$
IOPS=\frac{5000 MB}{4 KB}≈1,250,000 IOPS
$$
**当传输大块数据 (1MB) 时：**
$$
IOPS=\frac{5000 MB}{1 MB}=5,000 IOPS
$$


当带宽（分子）已经被跑满固定住时，分母（块大小）越大，结果（IOPS）自然就变得越小。数据块越大，每个 I/O 请求占用的“带宽资源”就越多，单位时间内能通过的数量自然就少了。



从微观的时间轴来看，处理一个 I/O 请求的时间（Latency）由两部分组成：
$$
总耗时=开销时间 (Overhead)+数据传输时间 (Transfer Time)
$$
主控解析命令、查找地址、准备电路的时间。这部分时间很短，且是个**固定值**（比如 10µs），不管你读 4KB 还是 1MB，这部分时间差不多。

数据真正通过 PCIe 通道和闪存接口传输的时间。这部分是**可变值**，与数据大小成**正比**。



**读 4KB (小块)**，主控开销 10µs，数据传输 1µs (极快)，**总耗时 11µs**，硬盘很快就能空出手来处理下一个请求 **IOPS 极高**。

**读 1MB (大块)**，主控开销 10µs，数据传输 250µs (因为 1MB 是 4KB 的 256 倍，传输时间很长)，**总耗时 260µs** 硬盘被这个巨大的数据块占用了很久 **IOPS 暴跌**。

块大小越大，单个 I/O 请求的**序列化延迟 (Serialization Latency)** 越高，导致单位时间内能完成的 I/O 次数减少。



#### 访问模式

**访问模式 (Access Pattern)**

- **随机 (Random)**：磁头需要跳跃（HDD）或 FTL 表需要不断重新映射（SSD）。这是最考验硬盘“硬实力”的模式。
- **顺序 (Sequential)**：数据连续存放。通常此时我们更关注吞吐量（MB/s）而非 IOPS。



#### 队列深度

**队列深度 (Queue Depth, QD)**

存储设备（无论是机械硬盘 HDD 还是固态硬盘 SSD）的速度远低于 CPU 和内存。如果没有队列，CPU 发出一个读写指令后必须等待（CPU停止，无法处理其他任务）硬盘处理完成才能发下一个指令（即同步 I/O），这将导致极极低的效率。



**队列机制允许 CPU 一次性发送多个请求给硬盘，然后继续处理其他任务（异步 I/O）。**

- **HDD（机械硬盘）：** 队列的核心技术是 **NCQ（Native Command Queuing，原生指令排序）**。由于磁头移动是物理动作，耗时最长。NCQ 允许硬盘控制器接收多个指令（通常最大 QD=32），然后重新排列执行顺序，以最小化磁头的物理移动距离（类似电梯算法），从而提高效率。
- **SSD（固态硬盘）：** SSD 不需要物理寻道，其优势在于**并发性**。SSD 内部有多个闪存颗粒（NAND Dies）和多个通道（Channels）。高队列深度允许控制器将不同的数据请求同时分发到不同的颗粒上并行处理，从而跑满内部带宽。



**队列和队列深度**

SATA 协议，支持 1 个队列，每个队列的最大深度为 32（最大 32 个命令）。

NVME 协议，支持 65,535 (64K) 个队列，每个队列的最大深度为 64K （ 64K 个命令）。

| 协议标准    | 队列数量 (Queues) | 每个队列的深度 (Queue Depth) | 总并发能力 (理论上限) |
| ----------- | ----------------- | ---------------------------- | --------------------- |
| AHCI (SATA) | 1个               | 32 个命令                    | 32 个                 |
| NVMe (PCIe) | 65,535 个 (64K)   | 65,536 个命令 (64K)          | > 40亿 个 (4 Billion) |





现在的硬盘测试软件（如 CrystalDiskMark 8.0+）通常会使用 Q 和 T 标记测试项目。

**[ SEQ1M Q8T1 ]** 或 **[ RND4K Q32T16 ]**

- **Q (Queue Depth) ：每个队列的深度**，指每一条排队通道里，同时有多少个指令在等待。
- **T (Threads) ：线程数/队列数**，指 CPU 开启了多少个线程同时向硬盘发指令。在 NVMe 协议下，通常 **1 个线程对应 1 个独立的硬件队列**。



**QD32 (Q32T1)**：**1 个队列**，这个队列里塞满了 **32 个指令**。

**Q1T32**：**32 个队列（1 个线程管理 1 个队列）** 同时工作，但每个队列里 **只有 1 个指令**。

**Q32T16**：**16 个队列（16 个线程管理 16 个队列，每个线程一个队列）** 同时工作，且每个队列都塞满 **32 个指令**。



“单队列 CPU 加锁”是传统存储协议（如 AHCI/SATA）在现代多核 CPU 架构下最大的性能瓶颈。

SATA 硬盘（由于 AHCI 协议限制）**只有一个**命令队列（Submission Queue）。当操作系统运行繁忙的任务（如数据库）时，可能 16 个核心同时都想往这个唯一的队列发送IO请求。

如果不加控制，核心 A 正在写第 1 个指令的位置，写了一半；核心 B 突然插进来，也往第 1 个位置写数据。会造成内存数据覆盖、指令错乱、系统崩溃（蓝屏）。这在计算机科学中叫 **“竞态条件” (Race Condition)**。

为了防止混乱，操作系统必须在队列入口处放一把“锁” (Lock/Mutex/Spinlock)。 哪个核心成功加锁，这个核心才能往队列里放东西。放完后，必须进行加锁，下一个核心才能加锁。

CPU 核心 A、核心 B、核心 C 同时产生了一个 I/O 请求。它们同时向那个唯一的队列进行加锁。假设核心 A 手快，加锁成功。**核心 A** 进入“临界区” (Critical Section)，开始往内存队列里写指令。核心 B 和 核心 C 必须等待，不能进行进程切换（在自旋锁 Spinlock 机制下）。它们必须在原地**空转 (Spinning)**，不断地循环查询锁状态（锁释放被其他核心释放，解锁），核心 B 和 C 的 CPU 占用率是 100%，但实际没有进行任何操作。





#### 访问延迟

很多工程师认为 IOPS 越高越好，但忽略了延迟。**高 IOPS 必须建立在低延迟的基础上才有意义。**

理解 QD 最专业的视角是引入 Little's Law（利特尔法则）在存储领域的变体。三者关系如下：
$$
IOPS = \frac{Latency (seconds)}{Queue Depth}
$$
增加 QD，IOPS 线性增加，延迟保持低位不变。在延迟保持不变（理想状态）的情况下，增加队列深度可以直接提升 IOPS。这就是为什么在跑分软件（如 CrystalDiskMark）中，高 QD 的测试结果往往远高于低 QD。

随着 QD 增加，硬盘最终会达到处理能力的极限。当延迟已经因为队列积压而开始上升时（延迟开始抬头），说明硬盘的**物理处理能力（后端）已经达到或接近饱和**。**这是最佳的性能点。**此时继续加大 QD，IOPS 不再增加，而后续排队的请求必须等待前面的处理完，导致**延迟（Latency）直线上升**。

性能调优的目标通常是寻找一个最佳性能点，即在延迟可以接受的范围内，最大化利用 QD 来获得最高 IOPS。



硬盘到了多少 QD，延迟就会开始提升取决于硬盘内部的**并行单元数量（通道数 × Die 的数量）**。

- **SATA SSD / 机械硬盘：**虽然 SATA 支持 QD32，但实际上超过 QD 4 以后，延迟就开始明显增加了，因为 SATA 只有一条物理通道，很容易拥堵。
- **消费级 NVMe SSD (如三星 990 Pro)：**消费级主控通常有 4-8 个通道。当并发请求超过 64 时，所有通道都跑满了，延迟开始线性上升。
- **企业级 NVMe SSD (如 Intel D7-P5510, Samsung PM1735)：**企业级主控极其强大（16+ 通道），且固件对高并发做了极深优化。它们甚至在 QD 256 时还能保持相对平稳的延迟。



对于高性能 NVMe，通常建议将 **Total QD 控制在 64 - 128 之间**。在这个区间，吞吐量（IOPS）接近峰值，而延迟还未起飞。一旦超过 128，IOPS 提升很小，但延迟会翻倍。



### 内部组件对性能的影响

一块高性能的固态硬盘（SSD）实际上就是一台微型的、高度专用的“计算机”。它有自己的 CPU（主控）、内存（DRAM 缓存）和存储单元（NAND Flash）。

要深入理解 SSD 的性能（IOPS、队列深度、延迟、带宽），我们需要剖析其内部的三大核心部件：**主控芯片 (Controller)**、**NAND 闪存颗粒 (NAND Flash)** 和 **DRAM 缓存 (DRAM Cache)**。



**主控芯片 (Controller)** 

主控是一颗嵌入式处理器（通常基于 ARM 架构），它是 SSD 的指挥中心。所有的读写请求、队列管理、垃圾回收（GC）、磨损均衡算法都由它运行。

高端 NVMe 主控通常拥有多核心（3-5 核），有些核心专门负责前端接口（处理主机命令），有些负责后端闪存管理。

**主控芯片（SSD Controller）是一个极其复杂的片上系统（SoC, System on Chip）**。**通道控制器（NAND Interface / Channel Controller）是主控芯片内部的一个“子部门”或“功能模块”，它并不作为独立的芯片存在于电路板上。**

**通道是 SSD 主控芯片和 NAND 闪存颗粒之间传输数据的“物理导线”**，SSD 内部只有一颗主控芯片，多颗 NAND 闪存颗粒，主控芯片和同一颗闪存颗粒间可以有多个通道进行连接。



**NAND 闪存颗粒（NAND Flash）**

**NAND Flash **是存放数据的地方。数据被存储在浮栅晶体管（Floating Gate）或电荷捕获（Charge Trap）单元中。

NAND Flash （闪存颗粒）是在 SSD 上肉眼看到的**那几块黑色的方块**。

在**闪存颗粒的内部**，存在多个 **Die（晶粒 / LUN）**，它是一片薄薄的硅片。芯片厂商会利用堆叠技术，把 **8 个甚至 16 个 Die** 一层一层的堆叠在闪存颗粒的内部。**Die 是执行命令的最小“独立”单位。** 主控可以同时给第 1 层的 die 和第 10 层的 die 发送指令，它们互不干扰（只要不在同一个通道总线上冲突）。

**一颗闪存颗粒（Package）只能连接到一个通道（Channel）上**，一颗封装好的黑色颗粒（Package），它的所有引脚（Pin）都是焊接在**同一个通道**上的。它内部所有的 Die（无论是 1 个还是 16 个），都必须**共享**这唯一的通道 A。

主控芯片给通过通道给 die 发送指令的速度是非常快的，但是 die 执行指令的速度是比较慢的（相比于发送指令的速度），

在利用 Interleaving（交错操作）技术提升效率的机制中，整个流程可以被描述为“串行传输，并行执行”。

首先，主控芯片占用物理通道的总线带宽，向 Die 1 发送写入指令与数据。这一过程属于**数据传输阶段**，此时通道总线处于被独占状态。当 Die 1 完成数据的接收与锁存（Latch）后，随即启动内部的**编程操作（Program Operation）**，将电荷写入存储单元。关键在于，一旦进入耗时较长的内部编程阶段，Die 1 便会立即**释放对通道总线的占用**，仅将自身的逻辑状态标记为“忙碌（Busy）”，从而使外部物理通道恢复为**空闲（Idle）状态**。

主控监测到通道总线回归闲置后，尽管 Die 1 仍处于内部写入过程中，主控仍可利用这一时间窗口，立即向同一通道上的 Die 2 发起新的数据传输。Die 2 在接收完数据后，同样释放总线并开始执行其内部写入任务。

最终，虽然物理通道上的数据传输是**分时复用**的串行过程，但在微观时间尺度上，Die 1 与 Die 2 均处于**内部并发工作**的状态。这种机制有效地利用了“总线传输快”与“介质写入慢”之间的时间差，极大地掩盖了闪存颗粒物理写入延迟较高的特性，从而显著提升了整体吞吐量。



### 吞吐量

**吞吐量 (Throughput)**，在存储领域常被称为“带宽” (Bandwidth) 或“传输速率” (Transfer Rate)，它是衡量硬盘在**单位时间内搬运数据总量**的指标。

如果说 IOPS 衡量的是硬盘处理请求的速度，那么吞吐量衡量的就是硬盘传输速度（每秒传输多少数据）。



在底层物理层面，吞吐量并非独立存在，它是由 IOPS 和块大小决定的：
$$
吞吐量 (MB/s) = IOPS × Block Size (块大小)
$$
**小块瓶颈在 IOPS**

当你读写 4KB 小文件时，即便 NVMe 接口带宽有 8GB/s，硬盘也跑不满。因为 IOPS 到了极限（比如 20k IOPS），吞吐量就被锁死在 20,000×4KB=80MB/s。



**大块瓶颈在带宽**

当你读写 1MB 大文件时，IOPS 不需要很高，但接口带宽（如 SATA 的 600MB/s）或介质传输极限会成为天花板。



要测出硬盘的**最大吞吐潜力**，一般会从以下几个维度进行设置：

- **数据块**：**128KB** (Q1T1 常用) 或 **1MB** (测极限带宽常用)。块太小（如 4K）会让主控忙于处理中断和地址转换，无法喂饱传输通道。
- **访问模式**：**顺序续写**，如果是随机模式，HDD 需要寻道，SSD 需要查表，都会大幅降低吞吐。
- **队列深度**：**QD8 ~ QD32**，虽然顺序读写对 QD 并不敏感（QD4 可能就跑满了 SATA），但在测试 NVMe Gen4/Gen5 时，为了压榨出 7000MB/s+ 的速度，通常需要 **QD8 或 QD32** 配合多线程。



### 硬盘测试模式

如果要对一块硬盘进行**全面、专业**的 IOPS 性能“体检”，仅仅跑一个“4K 随机”是不够的。专业的测试（如企业级 POC 测试或深度评测）通常会涵盖 **“低队列（响应）”、“高队列（吞吐）”** 和 **“混合读写（实战）”** 三大场景。



**QD1 低队列测试**

这是最能反映**个人电脑、工作站、操作系统流畅度**的指标，也是最硬核的指标（因为它无法靠堆并发来作弊）。

**测试组合**：**块大小: 4K | 访问模式: 随机读 | 线程/队列数量: 1 | 队列深度: 1**



**QD32+ 高队列饱和测试**

测试硬盘的极限 IOPS（极限吞吐），也是验证厂商宣传单页 IOPS 数值是否虚标的测试，也是衡量**并发能力**的指标。

**测试组合**：**块大小: 4K | 访问模式: 随机读和写（分开测） | 线程/队列数量: 8~16 | 队列深度: 32**

摸底硬件上限，把 SSD 的控制器和通道全部跑满，看它最大能处理多少吞吐。高负载虚拟化服务器、双十一期间的电商数据库。

**注意**：对于 SATA 盘，QD32 是极限；对于 NVMe 盘，可能需要加到 QD128 甚至 QD256 才能压榨出极限性能。



**70/30 混合读写测试**

这是**最专业、最容易测试消费级 SSD 缺陷**的测试。因为在现实世界（特别是数据库）中，很少有纯读或纯写的情况。

**测试组合**：**块大小: 4K (或 8K) | 访问模式: 混合 (70% 读/ 30% 写) | 线程/队列数量: 8~16 | 队列深度: 32**

模拟数据库负载场景：SQL Server 或 MySQL 的典型负载大约就是 7:3 或 6:4 的读写比。

闪存介质在同时进行读和写时，主控需要频繁调度，且面临读写冲突。很多标称“读 50万、写 40万”的盘，一测混合 70/30，IOPS 直接掉到 10万以下。

对于不同的业务，有时也会测 **8K 70/30**（Oracle 数据库默认页大小是 8K）。



**不同块大小的扫描测试**

为了摸清硬盘在不同大小文件下的表现，通常会做一个 **Sweep (扫描)** 测试。

**测试组合：块大小：512B, 4K, 8K, 16K, 32K, 64K, 128K(分开测) | 访问模式：读和写(分开测) | 线程/队列数量：1 | 队列深度：32**

寻找**最佳 IOPS**：通常在 4K。

寻找**最佳带宽**：通常在 128K 或 1M。

观察曲线。优秀的硬盘，随着块大小增加，IOPS 会平滑下降，带宽平滑上升。劣质硬盘可能会在某个特定块大小（比如 16K）出现性能骤降（固件Bug或优化不足）。





## 磁盘阵列

RAID技术将多个独立的硬盘组合成一个逻辑单元，以提高存储性能、数据冗余或两者兼顾。常见的RAID级别包括：



**RAID 0 (条带化)**：

- **原理**：将数据分散写入多个硬盘，并行读写。
- **优点**：读写速度最快，容量利用率最高（所有硬盘容量之和）。
- **缺点**：无数据冗余，任何一块硬盘损坏都会导致所有数据丢失。



**RAID 1 (镜像)**：

- **原理**：将数据同时写入两块（或更多块）硬盘，形成镜像备份。
- **优点**：数据安全性最高，一块硬盘损坏数据仍可从另一块恢复，只要有一块硬盘未损坏数据就不会丢失。
- **缺点**：磁盘利用率最低（仅为总容量的一半），成本较高。



**RAID 5 (带奇偶校验的条带化)**：

- **原理**：将数据和奇偶校验信息分散写入至少三块硬盘，奇偶校验信息用于数据恢复。
- **优点**：兼顾性能和数据冗余，允许一块硬盘损坏而不丢失数据，磁盘利用率较高（N-1块硬盘的容量）。
- **缺点**：写入性能相对RAID 0低，需要进行奇偶校验计算。



**RAID 6 (双奇偶校验的条带化)**：

- **原理**：在RAID 5的基础上增加了一组奇偶校验信息，允许同时两块硬盘损坏而不丢失数据。
- **优点**：数据安全性比RAID 5更高。
- **缺点**：写入性能更低，需要更多磁盘，磁盘利用率略低于RAID 5。



**RAID 10 (RAID 1+0)**：

- **原理**：先进行RAID 1镜像，再对镜像组进行RAID 0条带化。
- **优点**：兼具RAID 0的高性能和RAID 1的高数据安全性，允许多块硬盘损坏（只要不同镜像组）。
- **缺点**：成本最高，磁盘利用率最低（仅为总容量的一半）。





## 硬盘接口和传输协议

### 硬盘物理接口

硬盘物理接口是计算机存储子系统中的核心组件，它定义了存储设备与主机控制器之间进行机械连接、电气信号传输以及逻辑协议通信的标准规范。接口的设计直接决定了数据的传输带宽、信号的稳定性、设备的扩展能力以及物理兼容性。

硬盘接口的功能主要涵盖三个层面：物理连接层面规定了接口的尺寸、针脚定义（Pinout）和防呆设计，确保硬件的物理匹配；电气层面定义了电压、电流标准及信号电平特性，保障数据在传输过程中的完整性；逻辑协议层面则涉及指令集的传递方式（如 ATA 指令集或 SCSI 指令集），决定了主机如何向硬盘发送读写请求并处理返回数据。



以下是对各类硬盘物理接口技术的深入解析，特别是针对业界容易混淆的 NGFF、M.2 与 NVMe 之间的关系进行了专门阐述。



#### IDE

**IDE 接口 (Parallel ATA)**

IDE 接口，在技术规范中正式名称为 PATA（Parallel ATA，并行 ATA），是早期计算机系统中占据主导地位的硬盘传输标准。IDE 接口采用并行传输技术，即在同一时钟周期内，通过多条数据线同时传输多个比特的数据。

由于并行传输在提升频率时会面临信号线之间的严重串扰（Crosstalk）以及信号同步偏移（Skew）问题，IDE 接口的带宽提升极为困难。其发展的巅峰仅达到 133 MB/s 的理论传输速率（ATA/133 标准）。在物理结构上，IDE 接口使用宽大的 40 针或 80 针扁平排线，这种设计不仅占据了大量的内部空间，严重阻碍了机箱内部的散热气流，而且缺乏热插拔（Hot Swap）功能的支持。此外，IDE 设备需要通过物理跳线（Jumper）来设定“主设备”（Master）和“从设备”（Slave）模式，这种繁琐的硬件配置方式不符合现代计算机即插即用的设计理念，因此目前已被完全淘汰。



#### SCSI

**SCSI 接口 (Small Computer System Interface)**

SCSI 接口曾是服务器和工作站存储的基石。与面向消费级的 IDE 不同，SCSI 拥有一套独立且复杂的指令集系统，且配备独立的控制芯片。这使得 SCSI 设备在进行数据读写时，能够极大程度地减少对主机 CPU 资源的占用（Low CPU Utilization），非常适合多任务并发处理的服务器环境。

传统的并行 SCSI 接口支持菊花链（Daisy Chain）连接方式，允许在一条总线上挂载多个设备。虽然并行 SCSI 的物理接口因体积庞大和信号干扰问题已不再使用，但 SCSI 指令集（SCSI Command Set）因其强大的管理和纠错能力，至今仍被封装在 SAS、USB 甚至光纤通道等现代接口协议中继续使用。



#### SAS

**SAS 接口 (Serial Attached SCSI)**

SAS 接口是 SCSI 技术的串行化进化版本，目前是企业级存储领域的主流标准。SAS 采用点对点（Point-to-Point）的串行传输方式，利用扩展器（Expander）可以连接成百上千个存储设备。

SAS 的核心优势在于高可靠性和强大的纠错能力。物理设计上，SAS 接口支持双端口（Dual Port）技术，允许一个硬盘同时连接到两个不同的控制器，实现多路径输入输出（Multipath I/O）。若一条路径发生故障，系统可无缝切换至另一条路径，确保业务连续性。此外，SAS 接口在背板设计上向下兼容 SATA 硬盘，即 SAS 控制器可以识别并运行 SATA 硬盘（利用 STP 协议将 SATA 指令封装在 SAS 帧中），但 SATA 控制器无法识别 SAS 硬盘，这种单向兼容性为企业分级存储提供了灵活性。



#### SATA

**SATA 接口 (Serial ATA)**

SATA 接口的出现标志着存储传输技术从并行向串行的根本性转变。SATA 采用差分信号传输（Differential Signaling）技术，虽然在一个时钟周期内仅传输一位数据，但通过极大地降低信号干扰，从而能够实现极高的工作频率，最终获得远超并行接口的总带宽。

SATA 接口经历了多次迭代，目前的 SATA Revision 3.0 标准提供了 6 Gbps 的理论带宽（实际吞吐量约为 600 MB/s）。在逻辑层面，SATA 引入了 AHCI（Advanced Host Controller Interface，高级主机控制器接口）规范，支持 NCQ（Native Command Queuing，原生指令队列）技术，允许硬盘内部优化读写指令的执行顺序，从而大幅提升随机读写性能。物理上，SATA 接口采用 L 型 7 针数据接口和 15 针电源接口，支持点对点连接，消除了主从盘设置的复杂性，并全面支持热插拔功能，是目前机械硬盘和 2.5 英寸固态硬盘的标准接口。



#### ｍSATA

**mSATA 接口 (Mini-SATA)**

mSATA 接口是 SATA 国际组织（SATA-IO）为了适应笔记本电脑及小型化设备需求而制定的微型化接口标准，遵循 MO-300 规范。

mSATA 在电气特性和逻辑协议上与标准的 SATA 接口完全一致，同样受限于 SATA 3.0 的 6 Gbps 带宽瓶颈。不同之处在于其物理形态，mSATA 摒弃了传统硬盘的外壳，直接以 PCB 模块的形式插入主板上的 Mini-PCIe 插槽（但在电气定义上是 SATA 通道而非 PCIe 通道）。由于 mSATA 无法利用 PCIe 通道的高速特性，且占用的物理空间相比后续标准依然较大，它已逐渐成为一种过渡性技术，主要存在于旧款便携设备中。



#### M.2（NGFF）

**M.2 接口 (原名 NGFF) 与 NVMe 的关系解析**

M.2 的早期名称为 NGFF，NGFF 的全称是 **Next Generation Form Factor (NGFF)** ，后被正式命名为 M.2，它是一种紧凑型接口**标准/规格**，设计用于小型设备，如笔记本电脑、超极本和台式机主板。



在讨论 M.2 时，常常伴随着 NGFF、NVMe 和 SATA 等术语的混淆。需要明确的是，**NGFF（Next Generation Form Factor）是英特尔主导开发该标准时的项目代号，正式标准化后的名称即为 M.2。** 因此，NGFF 就是 M.2，两者指代的是同一种物理接口形态标准。

**M.2（原 NGFF）主要支持 SATA 和 PCIE 总线协议，早期（名称为 NGFF 时）主要使用基于 SATA 总线的 AHCI 协议，后期（正式命名位 M.2 时）主要用于基于 PCIE 总线的 NvMe 协议。所以很多时候会被混淆为 NGFF 支持 SATA 协议，M.2 支持 NVME 协议。**

**关于“NGFF 是 SATA 协议，M.2 是 NVMe 协议”的说法是错误的。** M.2 仅仅是一个物理接口和尺寸标准（Form Factor），它本身并不绑定某一种特定的传输协议。

M.2 接口可以支持多种通道和协议： 当 M.2 接口走 **SATA 通道**时，它传输的是 AHCI 协议，此时其性能上限与普通 2.5 英寸 SATA 固态硬盘完全相同（最高 600 MB/s），这种 M.2 硬盘主要用于兼容旧系统或低成本方案。 当 M.2 接口走 **PCIe 通道**时，它通常搭配 **NVMe (Non-Volatile Memory express)** 协议。NVMe 是一种专门为闪存存储设计的逻辑设备接口规范，旨在利用 PCIe 通道的高带宽（如 PCIe 4.0 x4 可达 7000+ MB/s）和低延迟特性，大幅降低 CPU 的指令开销并支持极高的队列深度。

因此，M.2 是物理形态，SATA 和 PCIe 是数据传输通道，AHCI 和 NVMe 是逻辑层协议。一块 M.2 接口的固态硬盘，既可能是慢速的 SATA 协议盘，也可能是高速的 NVMe 协议盘，区别在于其金手指的缺口设计（B Key 通常对应 SATA 或 PCIe x2，M Key 对应 PCIe x4）以及主控芯片的支持。



为了防止用户把错误的设备插错插槽，M.2 接口设计了不同的缺口（Key）。对于 SSD 来说，最重要的两个是：

- **M Key (Socket 3)：** 只有一个缺口（位于右侧）。这是 **NVMe SSD** 的标准形态，支持 PCIe x4 通道，速度最快。
- **B Key / B+M Key (Socket 2)：** 有两个缺口或一个位于左侧的缺口。通常用于 SATA 协议的 M.2 硬盘，或者是速度较慢的 PCIe x2 NVMe 硬盘。



**注意：** 购买 M.2 NVMe 固态硬盘时，你通常买到的都是 **M Key**（单缺口）。



用户最常看到的数字（如 2280, 2242）。这组数字并非随机代码，而是严格代表物理尺寸（单位：毫米）：

**前两位数字（宽度）：** 绝大多数台式机和笔记本的 SSD 宽度统一为 **22mm**。

**后两位/三位数字（长度）：** 代表 SSD 的长度。

- **2280：** (22mm x 80mm) —— **最主流的标准**，用于台式机和大多数笔记本。
- **2230：** (22mm x 30mm) —— 极短，常用于 Steam Deck、Surface Pro 等紧凑型设备。
- **2242 / 2260 / 22110：** 较少见，分别用于特定工业设备或服务器。



#### U.2

U.2 接口 (SFF-8639)

U.2 接口最初被称为 SFF-8639，是专门为企业级数据中心设计的高性能接口标准。它的设计初衷是解决 M.2 接口在服务器环境中散热困难且不利于热插拔维护的问题。

U.2 接口在物理形态上通常用于连接 2.5 英寸规格的固态硬盘（看起来像加厚的传统笔记本硬盘），但其接口端非常密集，整合了 PCIe x4 通道、SATA 通道以及 SAS 通道的信号引脚（虽然主要用于 PCIe）。U.2 接口让固态硬盘能够像传统 SAS 硬盘一样支持前端热插拔，同时利用 PCIe x4 通道和 NVMe 协议提供极高的吞吐量。它结合了 M.2 的高性能和 SAS 的易维护性，是目前高性能服务器存储的主流选择。



### 数据传输通

传输通道决定了电气信号的传递方式、物理带宽的上限以及信号的完整性设计。



#### PATA

Parallel ATA (PATA) 通道

这是 IDE 接口背后的传输技术。其核心特征是**并行传输**，即在一条宽排线上，同时有 16 位（16-bit）数据并行流动。虽然这种方式在低速时效率尚可，但随着频率提升，相邻信号线之间产生的电磁干扰（串扰）以及不同线路信号到达时间不一致（信号偏移）的问题呈指数级放大。这导致 PATA 通道的频率难以提升，最终止步于 133 MB/s 的物理带宽极限。此外，PATA 通道不支持全双工通信，即同一时间只能进行读或写，不能同时进行。



#### SATA

Serial ATA (SATA) 通道

SATA 通道是为了解决 PATA 的物理瓶颈而设计的。它放弃了并行传输，转而采用**差分信号（Differential Signaling）**技术的串行传输。SATA 通道只有两对数据线（一对发送，一对接收），通过极高的时钟频率来换取带宽。差分信号具有极强的抗干扰能力，使得 SATA 3.0 的频率可以轻松达到 6 GHz，提供 6 Gbps 的带宽。SATA 通道通常采用半双工模式（虽然物理上有收发两对线，但协议交互上通常受限于应答机制），且其带宽对于现代固态硬盘来说已成为瓶颈。



#### SCSI

Parallel SCSI 通道

这是早期企业级存储的通道标准。与 PATA 类似，它也是并行传输，但其总线宽度更宽（8 位或 16 位宽带），电气性能更强，支持更长的线缆和更多的设备挂载（菊花链）。Parallel SCSI 通道设计了复杂的总线仲裁机制，允许在同一条总线上存在多个控制器（Initiator）。然而，它同样未能逃脱并行传输的物理定律限制，因信号同步和干扰问题，最终被串行技术取代。



#### SAS

Serial Attached SCSI (SAS) 通道

SAS 通道是企业级存储的骨干。它继承了 SCSI 的逻辑优势，但在物理层面上采用了与 SATA 类似的串行技术。SAS 通道的一大突破是支持**全双工（Full-Duplex）**通信，这意味着数据可以在同一时刻双向传输（同时读和写），理论吞吐量翻倍。此外，SAS 通道支持**双端口（Dual Port）**技术，一个设备可以同时通过两条独立的 SAS 通道连接到两个不同的控制器，提供了物理链路的冗余。目前的 SAS-4 标准已能提供 24 Gbps 的单通道带宽。



#### PCIe

PCI Express (PCIe) 通道

PCIe 是目前最高速的通用输入输出通道，也是 NVMe 硬盘的物理基础。与 SATA 或 SAS 这种专门为存储设计的总线不同，PCIe 是直连 CPU 的系统总线。PCIe 采用**多通道（Lane）**聚合技术，每一个通道（x1）都是一对独立的收发差分线。M.2 或 U.2 硬盘通常使用 x4（4 个通道）连接。 PCIe 的优势在于极低的延迟和极高的扩展性。数据无需经过南桥或特定的存储控制器转换，直接直达 CPU 的内存子系统。PCIe 4.0 x4 可提供 64 Gbps (约 8 GB/s) 的带宽，而 PCIe 5.0 x4 则翻倍至 128 Gbps，彻底释放了闪存介质的性能。



### **逻辑协议**

逻辑协议是运行在传输通道上的“语言”，它定义了操作系统如何向硬盘发送指令（读、写、擦除、查询状态）以及硬盘如何反馈数据。



#### ATA

ATA 指令集 (ATA Command Set)

这是最古老的硬盘指令集，最初为 IDE 硬盘设计。它定义了最基础的寄存器操作和数据块传输命令。虽然物理接口已经进化了数代，但 ATA 指令集的影子依然存在。SATA 硬盘在启动阶段或兼容模式下，依然会模拟传统的 ATA 行为。



#### AHCI

AHCI (Advanced Host Controller Interface)

AHCI 是专门为 SATA 接口设计的硬件机制和协议规范。在 IDE 时代，CPU 需要通过繁琐的 PIO（可编程输入输出）模式干预数据传输，效率极低。AHCI 引入了 DMA（直接内存访问）机制，大幅降低了 CPU 占用率。 AHCI 最重要的贡献是引入了 **NCQ (Native Command Queuing，原生指令队列)**。NCQ 允许硬盘在内部维护一个深度为 32 的指令队列，硬盘控制器可以根据磁头的位置，智能地重新排序读写任务，减少机械臂的挥动次数，从而提升机械硬盘的随机读写性能。然而，AHCI 是为高延迟的机械硬盘设计的，对于零延迟的闪存设备，AHCI 繁琐的寄存器交互成为了性能累赘。



#### SCSI

**SCSI 指令集 (SCSI Command Set)**

SCSI 协议是企业级存储的通用语言。与 ATA 仅关注“读写扇区”不同，SCSI 协议是一套庞大的设备管理标准，它将存储设备抽象为逻辑单元（LUN）。SCSI 协议具有极强的容错性、错误报告机制和复杂的任务管理能力（如 Tagged Command Queuing）。 SCSI 协议的伟大之处在于其**传输层无关性**。它既可以运行在并行的 SCSI 线缆上，也可以封装在 SAS 帧中，甚至可以通过光纤通道（FC Protocol）或以太网（iSCSI）进行远程传输。它是现代 SAN（存储区域网络）的基石。



#### NVMe

**NVMe (Non-Volatile Memory express)**

NVMe 是专门为闪存（Flash Memory）和下一代非易失性存储设计的协议，它彻底抛弃了为机械硬盘设计的 AHCI 包袱。NVMe 协议建立在 PCIe 通道之上，其核心优势体现在以下几个维度：

- **并行性与队列深度：** AHCI 只有一个指令队列，深度为 32；而 NVMe 支持多达 64,000 个队列，每个队列深度可达 64,000。这完美契合了现代多核 CPU 和 SSD 内部多通道并发读写的特性。
- **低延时与精简指令：** NVMe 大幅简化了指令集，减少了 CPU 发送 I/O 请求时的寄存器读写次数。在 AHCI 中，一次读写可能需要多次 CPU 中断和寄存器交互，而在 NVMe 中，CPU 与硬盘通过共享内存直接交换指令，延迟降低了数个数量级。
- **直接内存访问：** NVMe 充分利用 PCIe 的 DMA 特性，数据直接在系统内存和 SSD 之间传输，无需 CPU 搬运，极大地释放了计算算力。



## 存储架构

### DAS

**DAS (Direct Attached Storage，直连存储/直接附加存储)**

DAS 是一种存储架构，指**存储设备**通过物理线缆（如 SATA、SAS、USB）或直接插槽（PCIe、M.2）**直接连接**到服务器或工作站，中间不经过任何交换机、路由器或网络节点。



在 DAS 架构中，存储设备（硬盘）是主机独占式访问的，这是 DAS 最本质的特征，也是区分它与 NAS/SAN 的金标准。连接 DAS 的那台服务器（主机），拥有对存储设备的**绝对控制权**。主机的操作系统（OS）直接管理硬盘的文件系统（如 NTFS, XFS, ext4）。

对于网络中的其他计算机来说，这块 DAS 硬盘是无法访问的（其他主机不知道这块硬盘的存在）。除非这台拥有硬盘的主机主动搭建一个文件共享服务（如 Windows 共享），否则其他人都无法直接访问里面的数据。



#### 属于 DAS 的物理接口类型

典型的 DAS 专属接口有 IDE, SATA, mSATA, M.2 (NVMe/SATA)，这些接口完全符合“存储设备直接通过线缆或插槽连接到主机服务器的总线上，中间没有任何网络交换设备”的定义。从这个定义出发，面向消费级和工作站的接口是纯粹的 DAS 接口。

IDE 和 SATA 接口的设计初衷就是“一对一”或“一对少”的短距离连接。它们利用主板上的南桥芯片或独立的 HBA（主机总线适配器）卡，直接与 CPU 通信。无论是机箱内的硬盘，还是通过 eSATA 连接的外部硬盘盒，其逻辑链路都是点对点的，操作系统直接管理文件系统。

M.2 和 mSATA 更是将 DAS 的概念极致化。它们直接插在主板上，利用 PCIe 或 SATA 通道直接与 CPU 交换数据。特别是 M.2 NVMe，它实际上是 PCIe 总线的延伸，让存储设备成为了系统内部的一个直接寻址的内存外围设备。因此，这类接口构建的存储系统，在物理和逻辑上都严格属于 DAS 范畴。





#### SAS 接口与“网络化”的界定

SAS 接口拥有强大的**SAS Fabric（交换结构）功能。通过 SAS Expander（扩展器）和 SAS Switch（交换机），成百上千块 SAS 硬盘可以组成庞大的存储池，并被多台服务器共享访问。**

虽然在物理层面上，硬盘依然是“连接”在某种背板上，但这种架构已经具备了小型 SAN 的特征。更重要的是，在大型企业级 SAN 存储阵列（如 EMC, IBM, Huawei 的高端存储）内部，其后端连接磁盘的物理接口通常也是 SAS。



虽然通过一根线连接主板到 SAS Expander（扩展器），再由扩展器连接数十块硬盘。这在**物理拓扑**上，确实构成了一个小型的“交换网络”。SAS 协议内部有 SAS Address（类似 MAC 地址），数据包在扩展器中通过路由表进行转发。

但是在**企业级存储架构（IT Architecture）**的定义中，**这种连接方式依然被称为 DAS**。



当多个主机通过 **SAS Switch（独立的 SAS 交换机）** 连接到同一个存储阵列，且存储阵列拥有自己的**智能控制器（Storage Controller）**，由控制器将硬盘虚拟化成 LUN（逻辑单元号）分发给不同主机，且主机之间需要进行复杂的路径仲裁时，这时的 SAS 架构才开始具备 SAN（存储区域网络）的特征。



**SAS 连接 Expander 确实用到了网络化技术，但在架构分类上，只要它是挂在某台服务器后端独享的扩展柜，它依然被归类为 DAS。**



#### U.2 接口

物理上，U.2 接口将 SSD 通过 PCIe 总线直接挂载在服务器 CPU 上，这是彻头彻尾的 DAS 物理连接。

逻辑上，正如您所说，这台服务器运行特定的软件（如 Ceph, VMware vSAN, MinIO），将这块本地 DAS 硬盘的容量池化，并通过以太网（TCP/IP 或 RDMA）对外提供服务。此时，对于**外部访问者**来说，这是一个网络存储；但对于**承载硬盘的那台服务器**来说，这依然是 DAS。

U.2 接口本身，是用于构建这种架构的物理积木，它本身是 DAS 接口。



#### SCSI 接口和 SCSI 协议

**SCS 接口和 SCSI 硬盘**

当老一辈工程师指着一个设备说“这是 SCSI 硬盘”时，他们指的是**并行 SCSI (Parallel SCSI)** 接口。

这是一种使用 50 针或 68 针粗大线缆的物理接口。它支持“菊花链”（Daisy Chain）连接，即一根线缆上可以串联 8 个或 16 个设备（包括硬盘、扫描仪、磁带机）。设备上需要通过跳线设置 ID（0-15）。 **在物理接口层面上，并行 SCSI 接口属于 DAS**。它直接连在主机的 SCSI 卡上。 

这种物理接口和线缆已经彻底被淘汰，被 SAS（串行 SCSI）物理接口取代。



**SCSI 协议**

这是 SCSI 最重要的遗产。SCSI 定义了一套通用的**语言（指令集）**，比如“读取扇区”、“写入数据”、“查询设备状态”。

这套语言设计得非常完美且与物理层无关。因此，现代存储接口虽然换了物理线缆，但内核依然在说 SCSI 语言：

- **SAS (Serial Attached SCSI)：** 物理上用串行线缆，逻辑上就是把 SCSI 指令打包，排队发送出去。
- **FC (Fibre Channel)：** 物理上用光纤，逻辑上是把 SCSI 指令封装在光纤帧里传输。
- **USB 移动硬盘：** 你可能不信，USB 大容量存储协议（UASP）底层传输的也是 SCSI 指令集。
- **iSCSI：** 物理上是以太网线，逻辑上是把 SCSI 指令封装在 TCP/IP 包里。

**所以，SCSI 协议本身不属于 DAS 或 SAN，它是数据沟通的通用语言。**



### NAS

NAS 本质上是一台经过优化的、专门用于提供文件共享服务的服务器。它拥有自己的操作系统和文件系统。NAS 通过以太网连接到网络中，客户端服务器不需要管理底层的磁盘扇区，而是直接通过网络协议请求“读写某个文件”。NAS 提供的是**“文件存储”**。它的核心优势在于极其方便的跨平台共享（Windows 和 Linux 可以同时访问同一个文件夹），但由于通过以太网传输文件流，且经过了多层协议封装，其延迟（Latency）通常高于 DAS 和 SAN，不适合高性能数据库应用。



### SAN

SAN 是一个独立于局域网（LAN）之外的专用高速存储网络。它将存储阵列（Disk Array）和服务器通过光纤交换机或专用以太网交换机连接起来。 在 SAN 架构下，服务器看到的不是一个“共享文件夹”，而是一块块“本地磁盘”。虽然这块磁盘物理上远在机房的另一端，但操作系统认为它是本地的未格式化磁盘。服务器需要自己对这块磁盘进行分区和格式化。SAN 提供的是**“块存储”**。它结合了 DAS 的高性能（低延迟、块级访问）和 NAS 的集中管理优势，是企业级核心数据库的标准配置。



### 分布式存储

分布式存储并非一种单一的硬件设备，而是一种通过软件将大量独立的物理存储节点（通常是通用的 x86 服务器及其内部的硬盘）互联起来，构建成一个逻辑上统一、资源上共享、性能上可线性扩展的存储系统的架构技术。它的核心理念是从“纵向扩展（Scale-up）”向“横向扩展（Scale-out）”的范式转移。

在传统的集中式存储（如双控 SAN 存储）中，所有的 I/O 请求都必须经过有限的控制器（通常是 2 个），硬盘被锁死在专用的盘柜里。当性能或容量不足时，用户往往只能购买更昂贵的控制器，这种模式被称为 纵向扩展（Scale-up）。而分布式存储打破了这一物理边界。它将数据打散，分散存储在通过高速以太网互联的多台服务器（节点）上。当需要提升性能或容量时，只需向集群中添加新的服务器节点，系统的总带宽、IOPS 和容量便会随之线性增长。这种架构使得存储系统能够轻松应对 PB 级甚至 EB 级的数据规模。



**元数据管理与数据分布**

分布式存储的大脑是**元数据（Metadata）管理。元数据记录了“数据块 A 存放在哪台服务器的哪块硬盘上”这一映射关系。根据元数据管理方式的不同，分布式存储主要分为两种流派。**

**一种是中心化元数据架构（如 HDFS）**，它拥有专门的元数据服务器（NameNode）来记录目录树和文件位置，这种方式管理简单，但元数据服务器容易成为性能瓶颈，限制了集群规模。

**另一种是更为先进的去中心化架构（如 Ceph 的 CRUSH 算法）**，它取消了查表操作，客户端通过计算哈希算法（Hash）直接算出数据应该落在哪个节点上。这种算法寻址方式彻底消除了元数据瓶颈，使得集群可以无限扩展。



**副本与纠删码技术**

既然数据分散在成百上千台普通的服务器上，而普通服务器的故障率远高于专用存储阵列，分布式存储如何保证数据不丢？它抛弃了传统的 RAID 技术（因为 RAID 重建时间太长，且受限于单机），转而采用**多副本（Replica）**和**纠删码（Erasure Coding, EC）**技术。

**多副本**是很多分布式块存储的默认策略。系统会将同一份数据自动复制成 3 份（三副本），并强制分散存储在不同的服务器，甚至不同的机柜中。只要集群中还有一份数据存活，业务就不会中断。当某块硬盘损坏，系统会利用剩余的副本，在全集群范围内并行恢复数据，其重构速度是传统 RAID 的几十倍。

**纠删码（EC）** 则多用于对象存储或冷数据场景。它利用复杂的数学算法（如里德-所罗门码），将数据分割成 N 个数据块，并计算出 M 个校验块，共 N+M 个块分散存储。只要损坏的节点数不超过 M，数据即可还原。EC 相比三副本极其节省空间（磁盘利用率可从 33% 提升至 70% 以上），但其计算开销较大，写入性能略低于多副本。



#### 软件定义存储

理解了分布式存储，再来理解软件定义存储（SDS）就水到渠成了。**分布式存储主要描述的是数据的物理分布架构（Topology），而软件定义存储（SDS）描述的是一种软硬解耦的交付模式与设计哲学。**

要深刻理解 SDS，必须透过现象看本质：SDS 的核心在于**解耦（Decoupling）**。在传统的存储阵列（如 EMC VMAX 或 HP 3PAR 旧型号）中，存储软件是“死锁”在专有的硬件控制器（ASIC 或 FPGA 芯片）上的。硬件和软件是一体的黑盒，用户无法将 A 厂商的软件运行在 B 厂商的硬件上。



SDS 打破了这种绑定。它将存储系统的**控制平面（Control Plane）**与**数据平面（Data Plane）**进行了彻底分离。

- **数据平面（硬件层）：** 变成了标准化的商品（Commodity）。SDS 允许使用通用的工业**标准服务器（x86 架构，如 Intel/AMD CPU）**和**标准的硬盘（SSD/HDD）**作为物理载体。硬件不再需要具备复杂的逻辑功能，只需负责最基础的 I/O 读写。
- **控制平面（软件层）：** 所有的智能、逻辑、策略管理全部上移至软件层。这套**软件是一个独立的操作系统或应用程序**，它负责将底层杂乱、异构的物理硬盘抽象成一个统一的“虚拟存储池”。诸如数据去重、压缩、快照、自动精简配置、异地复制等高级功能，完全由运行在 CPU 上的代码实现，而不再依赖专用的硬件电路。



SDS 不仅仅是“运行在 x86 上的存储软件”，它引入了**策略驱动（Policy-Driven）**的自动化管理理念。

在传统存储中，管理员创建一个卷（LUN），需要手动选择 RAID 级别、指定硬盘组。而在 SDS 架构中，管理员只需定义策略，如**“卷（LUN）的性能等级为高性能，要求 10,000 IOPS，三副本保护”**。SDS 软件会自动分析底层硬件资源，动态地在集群中调度最合适的 SSD 和节点来满足这一需求，并实时监控是否违背策略。

SDS 具备极强的**可编程性**。它对外暴露标准的 RESTful API 接口，这使得存储系统不再是一个静态的设备，而是一个可以被云平台（如 OpenStack, Kubernetes）或自动化脚本动态调用的服务。这也正是“基础设施即代码（Infrastructure as Code）”在存储领域的体现。



#### SDS 与分布式存储的关系

这是业界最容易混淆的两个概念。很多人将 Ceph 既称为分布式存储，又称为 SDS，这导致了概念的重叠。为了厘清它们的关系，我们需要从**“架构形态”**与**“交付模式”**两个维度进行剖析。

**分布式存储**

**分布式存储**实际上是一种**架构类型**，与其**相反**的架构类型是**集中式存储**。只要一个系统将数据分散存储在多个独立的物理节点上，并通过网络进行互联和协同，它就是分布式存储架构。这是一种物理拓扑的描述。



**软件定义存储**

**软件定义存储**实际上是一种**交付模式**，与其**相反**的交付模式是**软硬一体的专用存储**，只要存储软件可以独立于硬件安装、升级和迁移，并且能够通过软件定义资源池，它就是 SDS。这是一种产品形态的描述。



**SDS 通常采用分布式架构**

虽然定义不同，但在现代企业级市场中，**绝大多数优秀的 SDS 产品，其底层架构必然是分布式的。**

这是由硬件基础决定的。SDS 既然选择使用通用的 x86 服务器作为硬件，就必须面对一个现实。

通用服务器的可靠性远不如专用的存储阵列，单点故障率较高。为了在不可靠的通用硬件上构建出高可靠的企业级存储，SDS 软件必须采用分布式架构中的**多副本（Replication）**或**纠删码（Erasure Coding）**技术，利用多台服务器的冗余来抵消单台服务器的故障风险。

像 Ceph、VMware vSAN、Nutanix 乃至华为的 FusionStorage，它们既是标准的 **SDS**（软件独立交付，运行在通用 x86 上），同时也是标准的 **分布式存储**（多节点互联，横向扩展）。



**非分布式的 SDS**

需要深入的了解两者的区别，需要知道一个特例：**ZFS**。

当你在单台服务器上安装 TrueNAS（基于 ZFS 文件系统）来管理这台服务器内部的硬盘时，这完全符合 **SDS** 的定义（软件定义了 RAID，独立于硬件），但它并不是 **分布式存储**（数据没有跨节点）。



在当今的数据中心里，**分布式存储是实现 SDS 愿景的最佳物理架构**。SDS 赋予了存储系统灵活性、自动化能力和软硬解耦的商业优势，而分布式架构则为 SDS 提供了无限的扩展能力（Scale-out）和在通用硬件上实现 99.9999% 可靠性的技术底座。两者相辅相成，共同构成了现代云存储的基石。



### 磁带库

**磁带库 (Tape Library / Cold Storage)**

这是一种极其古老但至今无法被替代的架构，通常被称为“离线存储”或“近线存储”。 

它是机械臂和磁带机的组合。数据不在线存储在磁盘上，而是存储在磁带盒里。当需要读取数据时，机械臂将磁带抓取放入驱动器。 

它是归档架构的终点。由于磁带离线时完全不耗电且物理隔离（防黑客），它在长期冷数据归档和勒索病毒防护领域拥有独立且崇高的地位，不属于 DAS/NAS/SAN 的在线范畴。



### 统一存储

**统一存储 (Unified Storage)**

严格来说，这不算一种新的物理架构，而是一种产品形态的进化。

一台设备同时支持 SAN（FC/iSCSI）和 NAS（NFS/CIFS）协议。底层是一套物理硬盘，上层通过操作系统同时对外提供“块”和“文件”服务。目前市面上主流的中高端存储（如 NetApp, Dell EMC Unity）大多是统一存储架构，目的是简化管理。



## 数据存储类型

### 块存储

**块存储 (Block Storage)**

这是最底层、最原始的存储形态。存储设备将数据以固定大小的“块”（Block，例如 4KB 或 8KB）或者“扇区”进行管理，并赋予每个块一个唯一的地址（LBA）。

操作系统可以直接访问这些物理地址，没有任何文件系统的抽象层。它速度最快，开销最小。

**DAS** 和 **SAN** 提供的都是块存储。



### 文件存储

**文件存储 (File Storage)**

这是建立在块存储之上的一层抽象，通常**使用文件系统实现**。文件系统（File System）记录了文件名称、目录结构、权限属性等元数据，并将这些信息映射到底层的“块”上。用户不需要知道数据在哪个扇区，只需要知道文件路径（如 `/home/user/doc.txt`）。

**文件存储（文件系统）**对人类友好，易于共享，但因为多了一层文件系统转换，随机读写性能弱于块存储。

NAS 提供的是文件存储，连接到 NAS 服务器的客户端是以文件为单位进行操作的。



### 对象存储

**对象存储 (Object Storage)**

这是针对海量非结构化数据设计的新型存储形态。它彻底扁平化，没有复杂的目录树结构（文件夹）。

数据被存储为“对象”。一个对象包含三部分：**数据本身（Data）**、**元数据（Metadata，如作者、创建时间、标签）**、**全局唯一标识符（ID）**。用户通过 API（通常是 HTTP RESTful API）凭借 ID 来存取数据。

无限扩展性（Scale-out），适合存储 PB 级甚至 EB 级的数据，但不支持对文件内部进行随机修改（通常需要覆盖写入）。它不属于 DAS/NAS/SAN 的传统范畴，通常运行在通用的服务器集群上。



亚马逊 AWS 发布了 **Amazon S3（Simple Storage Service）**，亚马逊 S3 选择了互联网最通用的 **HTTP/HTTPS** 协议作为传输载体，并采用了 **RESTful API** 风格。在这一体系下，所有的存储操作被简化为几个核心的 HTTP 动词：PUT（上传/写入）、GET（下载/读取）、DELETE（删除）和 HEAD（获取元数据）。存储桶（Bucket）和对象键（Key）构成了扁平化的寻址逻辑。

这种设计的精妙之处在于它将存储接口从“系统内核态”提升到了“网络应用态”。开发者无需关心底层的磁盘扇区、RAID 级别或文件系统挂载，只需通过简单的网络请求即可在任何地点存取数据。S3 API 的出现极大地降低了对象存储的使用门槛，并展现了惊人的扩展性（Scalability）。

随着 AWS 的成功，OpenStack Swift（早期开源对象存储代表）和 Ceph（统一存储代表）等开源项目也相继涌现，但它们在接口层面最终都选择了兼容 S3 API。如今，当我们谈论对象存储的“标准化”时，指的是 **S3 兼容协议（S3 Compatibility）**。无论是私有云厂商（如 MinIO、Dell ECS）还是公有云服务商，其核心竞争力之一都在于对 S3 API 的兼容程度。



### KV 存储

**键值存储（KV 存储）**

虽然它常被归类为“对象存储”的简化版或“NoSQL 数据库”的一种，但在底层数据组织上，它是一种独立的类型。

数据没有复杂的文件结构，也没有元数据，只有简单的“键（Key）”和“值（Value）”对应关系。

它通常作为大规模分布式系统的底层缓存层（如 Redis, Memcached 的持久化层）或对象存储的底层索引引擎。



### 内存语义存储

**内存语义存储 / 字节寻址存储 (Byte-Addressable Storage)**

这是存储技术的最前沿，模糊了“内存”和“存储”的界限。传统的块存储最小读写单位是 4KB，而字节寻址存储允许像访问内存一样，直接读取或修改某一个具体的**字节（Byte）**，甚至是比特位。



**字节寻址 (Byte-Addressable)：** 这是指数据的访问粒度。

- **传统硬盘（块存储）：** 哪怕你只想修改一个字，硬盘也必须读写整个 4KB 的数据块（Block）。这就好比你要修改书架上的一本书，却必须把整个书架搬下来，改完再搬回去。这就是“块寻址”。
- **字节寻址：** 就像 CPU 访问内存（RAM）一样，CPU 可以直接通过内存地址，精确地只读取或修改**一个字节（Byte）**的数据，而不需要搬运无关的数据。这就是“内存语义”。



**持久内存 (PMem)** 是实现上述逻辑的**物理硬件**。它是介于 DRAM（内存）和 SSD（固态硬盘）之间的第三类存储层级。

它具备两个核心 DNA：

1. **像内存一样快（Memory-like）：** 它插在主板的内存插槽（DIMM）上，直接走内存总线（DDR总线）与 CPU 通信，延迟极低。
2. **像硬盘一样存（Storage-like）：** 它使用特殊的非易失性介质（如英特尔的 3D XPoint 相变存储技术），断电后数据不会丢失。

它在物理外观上与普通的内存条（DRAM）几乎一模一样。 它同样有金手指，同样插在服务器主板的 DDR4/DDR5 插槽里，同样带有散热马甲。如果不看标签，你很难凭肉眼区分它是普通的 RAM 还是 PMem。最著名的商用产品是 **Intel Optane DC Persistent Memory（傲腾持久内存）** 系列（注：虽然英特尔近期调整了业务，但这依然是该技术的教科书式代表）。



这种类型通常依赖于 **SCM (Storage Class Memory，存储级内存)** 技术，例如 Intel 的 Optane（傲腾）持久内存。

它比 NVMe SSD 还要快一个数量级，通常用于极高性能数据库的日志区或金融高频交易系统。它打破了传统“块存储”的 I/O 路径，CPU 直接通过内存总线访问数据。



## 存储网络传输协议

### 块级存储网络协议

块级存储网络协议是用于在网络上**传输块级数据**的通信标准。与文件级存储（例如，通过NFS或SMB共享文件）不同，块级存储将数据视为原始的、固定大小的数据块，不包含任何文件系统或目录信息。服务器通过这些协议访问存储设备时，会将远程存储视为本地连接的硬盘驱动器。



**iSCSI (Internet Small Computer Systems Interface)**

- **描述：** 这是最常见和广泛使用的通过标准以太网（TCP/IP）网络共享块设备的协议。它将 SCSI 命令封装在 TCP/IP 数据包中进行传输。



**Fibre Channel (FC)**

- **描述：** 严格来说，光纤通道本身是一种专用的高速网络技术，主要用于构建传统的 SAN。它并非在现有以太网基础设施上运行，而是需要专用的光纤通道交换机和主机总线适配器（HBA）。



**FCoE (Fibre Channel over Ethernet)**

- **描述：** 旨在将光纤通道的优势与以太网的灵活性结合起来。它将光纤通道帧封装在以太网帧中，允许光纤通道流量在支持数据中心桥接 (Data Center Bridging, DCB) 技术的以太网网络上运行。



**NVMe-oF (NVMe over Fabrics)**

**描述：** 这是为利用 NVMe SSD 的极致性能而设计的新一代网络存储协议。它通过多种网络传输（"Fabrics"）来承载 NVMe 命令，例如：

- **NVMe/TCP：** 通过标准 TCP/IP 以太网。
- **NVMe/RDMA：** 通过支持 RDMA（Remote Direct Memory Access）的网络，如 InfiniBand、RoCE (RDMA over Converged Ethernet) 或 iWARP。
- **NVMe/FC：** 通过光纤通道网络。



### 文件共享网络协议

文件共享网络协议是计算机网络中用于在不同节点（计算机、服务器、移动设备）之间传输、访问和管理文件的“通用语言”。这些协议定义了客户端如何向服务器发送请求（如打开、读取、写入、关闭文件），服务器如何响应，以及如何处理身份验证、权限控制和并发冲突（锁机制）。



#### SMB/CIFS

**SMB** 是目前世界上最广泛使用的文件共享协议，它源于 IBM，但由微软发扬光大，成为了 Windows 生态系统的核心命脉。可以说，你在局域网中使用的“网上邻居”或“映射网络驱动器”，背后运行的几乎都是 SMB 协议。

SMB 是一种**客户端-服务器（Client-Server）**请求响应协议。它不仅支持文件共享，还支持打印机共享和进程间通信（IPC）。SMB 的一个显著特点是其**有状态（Stateful）**特性：服务器会维护客户端连接的状态信息。这意味着如果网络中断，连接需要重新协商。SMB 拥有极其完善的**文件锁定（File Locking）**机制，支持“机会锁（OpLock）”，这允许多个用户同时编辑同一个文件而不产生数据冲突（例如在公司共享盘上打开 Excel 表格），这是许多其他简单传输协议无法做到的。



#### NFS

NFS (Network File System)

**NFS** 就是 Unix/Linux 世界的官方语言。它由 Sun Microsystems 开发，旨在让访问网络文件像访问本地文件一样透明。

**NFSv3:** 这是一个**无状态（Stateless）**协议。服务器不记录客户端打开了哪些文件。如果服务器重启，客户端甚至感觉不到（因为没有连接状态丢失），只要服务器恢复，客户端重试请求即可。但 NFSv3 安全性较弱，主要依赖 IP 地址进行信任验证，且使用动态端口，对防火墙极其不友好。

**NFSv4:** 这是一个巨大的飞跃，变成了**有状态（Stateful）**协议。它引入了单一 TCP 端口（2049）设计，便于穿透防火墙。NFSv4 极大地增强了安全性（支持 Kerberos 验证）和锁机制。



#### FTP

FTP 家族 (File Transfer Protocol 及其变种)

FTP 是互联网历史上最古老的协议之一，专为在不可靠的网络上传输文件而设计。与 SMB/NFS 这种“挂载”型协议不同，FTP 是纯粹的“传输”型协议。

**FTP (标准版)**: 它使用两个独立的 TCP 连接：**一个控制连接（端口 21）用于发送指令，一个数据连接（通常端口 20 或随机端口）用于传输实际数据。这种双通道设计导致了著名的“NAT 穿越”问题，分为主动模式（Active）和被动模式（Passive）。由于标准 FTP 采用明文传输用户名、密码和数据，极易被抓包窃取，在公网环境中已属于高危协议。**

FTPS (FTP over SSL): 这是在传统 FTP 协议之上包裹了一层 SSL/TLS 加密层（类似于 HTTP 变成 HTTPS）。它保留了 FTP 的双通道特性和复杂的端口协商机制，虽然安全，但在防火墙配置上依然繁琐。



SFTP (SSH File Transfer Protocol): 这是一个常被混淆的概念。 SFTP 不是 FTP 的加密版，它与 FTP 协议在底层逻辑上毫无关系。SFTP 是 SSH (Secure Shell) 协议的一个子系统。它使用单一端口（默认 22），所有数据（包括认证、控制指令、文件内容）都通过一个加密的 SSH 隧道传输。SFTP 继承了 SSH 的高安全性，配置简单（只需开一个端口），是目前 Linux 服务器管理和公网文件传输的首选标准。



#### WebDAV

**WebDAV (Web Distributed Authoring and Versioning)**

WebDAV 是一个非常独特的协议，它是 **HTTP 协议的扩展**。简单来说，它试图把“只读”的万维网变成“可读写”的存储空间。

WebDAV 在标准的 HTTP 方法（GET, POST）之外，增加了 PUT（上传）、DELETE（删除）、MKCOL（创建集合/目录）、COPY、MOVE 以及至关重要的 **LOCK/UNLOCK**（锁定）方法。 由于它是基于 HTTP/HTTPS 的，WebDAV 具有极强的**网络穿透性**。只要能上网（通过 80 或 443 端口），就能连接 WebDAV。这使得它成为公网远程访问 NAS 或云盘的最佳协议。许多应用（如 Word、Excel、Keepass、Obsidian）都内置了 WebDAV 支持，允许用户直接通过互联网编辑远程服务器上的文件，就像操作本地文件一样。



**S3 API (对象存储传输协议)**

虽然在“存储架构”中 S3 属于对象存储，但在“网络传输协议”的语境下，**S3 API（基于 RESTful HTTP）** 实际上已经成为了一种新形态的文件共享协议。

现代应用程序（如网盘客户端、备份软件、大数据分析工具）不再使用 SMB 或 NFS 去读取数据，而是直接通过 HTTP 发送 S3 API 指令（PUT Object, GET Object）来存取数据。虽然它不具备传统文件协议的“部分修改”或“POSIX 锁”特性，但凭借其极简的架构和无限的广域网扩展能力，它正在成为云原生时代的数据传输标准。



### 集群文件系统

**集群文件系统**（Cluster File System，简称 CFS）是一种特殊类型的文件系统，它允许多台计算机（即**集群**中的节点）同时访问和共享同一个存储设备上的数据。与传统的文件系统（如 NTFS 或 ext4）不同，传统文件系统通常只能由一台计算机直接访问，而集群文件系统设计的核心目标就是解决多节点并发访问数据的问题，同时确保数据的一致性和完整性。



当前**主流的存储架构正趋向于**通过**软件定义存储（SDS）**和**分布式存储**的理念，在**底层利用分布式存储**技术将**分散在多台通用服务器上的存储资源汇聚成一个弹性、可扩展的统一存储池**。这个强大的存储池**对外可以提供多种访问接口**：对于需要独占、高性能**块级访问**的客户端，它能通过类似于传统 SAN 的方式（例如利用 iSCSI 或 NVMe-oF）直接分配并呈现逻辑单元号（LUN）；而对于需要**多台主机并发访问共享文件数据**的场景，则可以在此**块级访问的基础上**，**叠加集群文件系统**来提供统一的文件命名空间和数据一致性保障，从而实现高效的文件共享与协作。这种分层且灵活的架构，使得存储系统既能满足多样化的应用需求，又能充分利用通用硬件，实现成本效益和卓越性能的平衡。
